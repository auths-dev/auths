# Red-Team Findings — Murmur Olm Backend + KERI↔Olm Join (2026-06-16)

**Scope:** the new message-encryption backend introduced by the vodozemac migration —
`crates/murmur-core/src/olm_backend.rs` and `src/channel.rs` (feature `olm`), the
`Cargo.toml`/`Cargo.lock` dependency wiring, and the KERI↔Olm join. **Not** a review of the whole
workspace; the audited library (vodozemac 0.10.0 / Olm) carries its own Least-Authority audit,
reconciled in §4.

**Engagement:** four parallel adversarial specialists (claims-auditor, crypto, protocol/join,
supply-chain), each told "treat green as guilty until proven honest," each adversarially verifying
(refute-first, PoCs in `/tmp`) before reporting. The author wrote both the implementation and its
tests, so a passing test was treated as evidence of nothing until independently confirmed.

---

## 1. Executive summary

**Posture: At-risk → Hardened (for the reviewed surface).** The custom crypto an external audit must
bless — the signed-bytes construction and `verify_rooted`'s signature/AID binding — was found
**sound**: injective signing bytes, no forgery, cross-AID and cross-protocol replay closed, rigor
matching the in-tree `prekey.rs`. The per-message crypto (v2 MAC pinning, downgrade protection, wire
codec, uniform errors, FS/PCS on the live session) is used **correctly** and is panic-free on hostile
input. No CRITICAL or HIGH confirmed.

Raised 18 candidate issues across the four specialists; after de-duplication and adversarial
verification, **6 substantive (all Medium/Low) + the supply-chain item** survived. Every actionable
one is **fixed** (commit `38533a5f`) or recorded as a risk. The top three real exposures were:
(1) the v2 full-MAC guarantee rode an *experimental* upstream feature and a loose version range (R9);
(2) encrypted-pickle persistence had no anti-rollback, defeating forward secrecy on the at-rest path
(R10/CR-1); (3) one one-time key with no fallback was a first-contact DoS with an opaque error (JN-3).

## 2. Attack surface map

| Boundary | What crosses it | What trusts it |
| --- | --- | --- |
| `OlmPrekeyBundle` (published) | recipient AID + Olm identity/OTK/fallback keys + KERI signature | `verify_rooted` (signature + AID + hygiene) before any session |
| first wire (prekey) | attacker-controlled bytes off the relay | `decode_wire` → `OlmMessage::from_parts` → `create_inbound_session` |
| `sender_olm_identity_key` arg | the sender's Olm identity key | **caller contract** — must come from the sender's KERI-verified bundle (JN-5) |
| message wire | tag byte + Olm ciphertext | `OlmChannel::decrypt` (full v2 MAC; uniform error) |
| encrypted pickle blob | at-rest session state | `from_encrypted_pickle` (pickle key; **no freshness** — R10) |

**Cleared (raised then refuted):** signing-bytes forgery/shuffle (JN-1 — injective, 0 collisions in brute force); downgrade bypass on normal messages (the session's pinned config governs, not the message's self-declared version); wire-tag confusion (rejected by inner version/protobuf shape); decryption oracle / panic on hostile bytes (0 panics across fuzzed inputs); feature-gating leakage (vodozemac absent from every default/production build); secret-bearing Debug/logs (only public key material is `Debug`; `OlmIdentity` is not `Debug`).

## 4. Upstream audit reconciliation — Least Authority, *Matrix vodozemac*, 2022-03-30

The Olm/Megolm ratchet rides vodozemac's public Least-Authority audit (10 issues A–J, 8 suggestions).
Most are library-internal and **Resolved** (zeroization A, input-validation B/Suggestion 4/6, overflow
D/E/F, dependency hygiene Suggestion 1, strict-Ed25519 Suggestion 5, test coverage Suggestion 7). Four
matter at *our* integration layer:

- **Issue J — MAC truncated to 64 bits (Unresolved upstream).** The audit flags the truncated MAC as
  a real authentication weakness ("home servers can perform such attacks", 2⁻⁶⁴ forgery per message).
  **In Murmur the untrusted relay is exactly that adversary.** Our decision to pin **v2 (full,
  untruncated MAC)** via `experimental-session-config` is the **direct remediation of audit Issue J** —
  this is the strongest justification for R9, not merely a precaution. (At audit time v2 did not exist;
  vodozemac later added it behind the experimental flag.)
- **Issue I — keys in memory vs swap/side-channel (Unresolved).** Cleartext identity/one-time/ratchet/
  MAC keys are extractable via swap or side-channel. → **R11.** Industry-standard for any software
  ratchet; bounded by FS/PCS + KERI revocation; encrypted at rest (R10); on iOS, app memory is not
  swapped to disk and the SE holds the AID root. Recorded as accepted in ENC-7.
- **Suggestion 8 — 40 skipped-message-key cap (Unresolved; Matrix declined config).** >40 out-of-order
  messages and further out-of-order decryption fails. → **R12.** Sharper for Murmur than Matrix: our
  adversarial store-and-forward relay can deliberately reorder/delay to force message-loss. We accept
  the bound (Matrix's misuse reasoning) but document it and rely on best-effort relay ordering + client
  gap detection. ENC-7 item.
- **Issue G — invalid inbound session removes unused OTK (Resolved upstream).** vodozemac now drops the
  OTK only *after* a successful decrypt, so an *undecryptable* prekey can no longer burn a victim's
  OTK. This already shrinks the OTK-exhaustion surface behind JN-3 (an attacker must produce a valid
  session to consume the key) — our fallback key (JN-3 fix) covers the remaining honest-contention case.

## 5. What an external audit (ENC-7) still owns

1. The **join** (§5 of the migration PRD): KERI resolution, bundle-signature verification, the
   Olm-identity↔AID binding, the `create_*_session` hand-off — confirmed sound here, but same-author.
2. The **inner-signature layer** that binds the sender's Olm key to the sender's AID (JN-5) — must be
   wired and mandatory before the channel carries traffic.
3. **At-rest storage** (R10/R11): rollback-protected, per-session pickle keys, the Keychain/SE binding.
4. The **out-of-order bound** (R12) against the real relay reordering model.
5. That **v2 is wired on both ends** (R9) and the experimental-feature dependency is tracked.

## 6. Adversarial test gaps now closed

Added this pass: `session_config_is_v2_full_mac`; real-FS later-state snapshot (both backends);
PCS positive control; OTK single-use + fallback recovery; message-ciphertext tamper; tampered-fallback
bundle rejection.

**Closed since:**
- **R10 (rollback)** — added `to_versioned_pickle`/`from_versioned_pickle`: a generation-bound pickle
  (key derived from `(pickle_key, generation)`) the storage layer can use to **reject a rollback**.
  `versioned_pickle_rejects_rollback_and_tamper` proves an earlier-generation blob is refused and a
  generation tampered upward fails to decrypt. (The storage layer must still persist the monotonic
  generation in rollback-protected storage — ENC-7 verifies that binding.)
- **R12 (out-of-order bound)** — `out_of_order_messages_within_the_skip_window_decrypt` locks in
  reordered delivery within Olm's 40-key skip window; the >40 drop is documented as a relay-ordering
  bound (R12) the transport must respect. A reorder fuzz *past* 40 against the real relay model stays
  an ENC-7 item.

All code-addressable findings are now fixed; the residue (the join, inner-signature binding JN-5, the
storage binding for R10, the reorder model for R12) is explicitly **ENC-7 / external-audit** scope.
