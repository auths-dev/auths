## The bar

After `auths init`, the product is **zero new verbs**:

```
git commit        # signed, trailered, done
auths verify HEAD # green
```

That's the whole contract. Every command we add to that loop is rent we charge on every
commit, forever, and developers will not pay it — they'll uninstall. Sigstore's `gitsign`
and plain SSH signing both achieve "commit as normal"; we do not get to be worse than the
incumbents at the exact thing we're replacing.

Today's reality, measured: **signing is green everywhere; verification is red
everywhere local.** A new developer following our own README signs their first commit
successfully and then fails to verify it — twice, for two different reasons. Meanwhile the
CI path (`--identity-bundle`) verifies everything. We built the enterprise path first and
left the laptop path broken. Launch-blocking.

---

## Decisions

### D1. `git commit` must produce a verifiable commit. `auths sign HEAD` is a repair tool, not the workflow.

**Today:** init sets `gpg.format=ssh`, `gpg.ssh.program=auths-sign`, `commit.gpgsign=true`,
and prints "Your next commit will be signed with Auths!" The commit does get an SSH
signature — but `auths verify HEAD` rejects it: *"Commit carries no Auths-Id/Auths-Device
trailer."* Structurally guaranteed: a `gpg.ssh.program` shim can only return signature
bytes; it cannot edit the message. Trailers are only added by `auths sign <ref>`
(`commands/sign.rs:65-110`), which **rewrites the commit**.

**Rejected:** documenting `git commit && auths sign HEAD`. Two commands per commit is two
too many — and worse than friction, it's a footgun: `sign HEAD` amends, so the SHA changes
after every commit. Sign after push and you're force-pushing. A workflow that rewrites
history as a matter of routine is disqualified on correctness grounds, not just UX.

**Rejected:** making verify accept trailer-less SSH-signed commits locally. That forks the
verification model — third-party verifiers still need the trailers, so we'd have commits
that verify on the author's machine and nowhere else. Worse than failing.

**The decision:** trailers go in at commit time, invisibly. `auths init` installs a
`prepare-commit-msg` hook that appends `Auths-Id`/`Auths-Device` (cheap: both values are
known locally; the gpg shim then signs over the trailered message). Use
`init.templateDir` plus hook installation on first `auths sign`/`verify` in existing
repos — and have `auths doctor` check for it. The hook is ~5 lines of shell. `auths sign
<ref>` survives only as the explicit repair/backfill tool for commits made before the hook
existed, and its docs say exactly that.

README and `signing-commits.md` already promise this UX. Keep the promise instead of
correcting the docs downward.

### D2. If I signed it, I can verify it. Self-trust is not optional.

**Today:** `auths sign artifact.txt` then `auths verify artifact.txt` on the same machine,
same `~/.auths`, fails: *"Failed to resolve public key from issuer DID… Use
--identity-bundle for stateless verification."* The issuer's KEL is sitting on disk; verify
just never looks. We literally tell the laptop user to use the CI workaround.

**The decision:** the verifier's local identity is always a trusted root for its own
verifications, resolved from local storage — no pinning, no flags, no bundle. This is not a
security relaxation: trusting keys you yourself control is the floor, not a concession.
Pinning exists for *other people's* roots. Any design where the author cannot verify their
own signature with zero arguments is broken by definition.

### D3. Signing in a repo pins your root in that repo. Nobody runs a pin command.

**Today:** even with trailers (post `auths sign HEAD`), verify fails: *"Root did:keri:E… is
not a pinned trusted root. Pin it in .auths/roots."* init pins your root only into the repo
you happen to be standing in when you run init (`init/mod.rs:378-388`) — i.e., for every
repo you'll ever actually work in, verification is off until manual ceremony. And the
suggested ceremony is unrunnable (D6).

**The decision:** the first `auths sign` (or hook-signed commit) in a repo auto-writes your
root into `.auths/roots` with a one-line notice. It's your repo and your signature; asking
permission to trust yourself is theater. The committed `.auths/roots` file is also how
teammates and CI inherit trust — so this one default silently sets up the *team* story too.
D2 covers you locally before the pin lands; the pin is for everyone downstream.

### D4. `init` is a transaction. Half an identity is worse than no identity.

**Today:** when init failed on the passphrase policy, it had already persisted the identity
record. The aftermath: `status` exits 0 and looks healthy, `sign` fails with jargon ("All
keys are rotation keys (--next- prefixed) or no keys exist"), `rotate` fails E5302, and
nothing anywhere suggests the actual fix (`auths reset --force`). Same disease family as
issue #250 (orphaned hardware keys at inception).

**The decision:** stage everything, commit at the end, roll back on any failure. A failed
init leaves the machine exactly as it found it. This is table stakes for a tool whose whole
pitch is integrity. No partial-state "doctor can fix it later" compromise — recovery flows
are where trust products go to die.

### D5. Errors name the input that caused them, in non-interactive mode especially.

**Today:** a weak `AUTHS_PASSPHRASE` produces `[AUTHS-E4203] storage error: key operation
failed… fix: Check keychain access and passphrase`. The real error — E3020 WeakPassphrase,
with the actual policy (≥12 chars, ≥3 of 4 character classes) — exists in the registry and
is never surfaced; the chain got flattened. In `--non-interactive` there's no prompt
fallback, so this vague wall is the entire experience. And init's "Checking prerequisites…
[OK]" ran *before* the failure without checking the one input that was fatal.

**The decision:** preflight the passphrase during the prerequisites step (init knows the
policy; check it before touching state — also what makes D4 cheap). Surface E3020, name the
source (`AUTHS_PASSPHRASE` env var), and print the policy inline. Generic "fix:" lines that
ignore the specific message above them (E5007 does this too) get deleted — a wrong hint is
worse than no hint.

### D6. Humans never type hex. Commands that suggest other commands must suggest runnable ones.

**Today:** the verify error prescribes `auths trust pin --did … --key <hex>`. Where does
`<hex>` come from? `whoami --json` → identity DID only. `key list --json` → aliases only.
`key export` → OpenSSH pem/pub/enc, no hex (and it takes `--passphrase` as a CLI argument,
which lands in shell history). The CLI's own remediation is unreachable through the CLI.

**The decision:** `trust pin --did <did>` resolves the key material itself — from the local
KEL, a `--bundle` file, or (later) the registry. The `--key <hex>` form survives for
air-gapped ceremony only. `whoami --json` returns the full identity surface (root DID,
device DID, key aliases + public keys) — it's also what every integrator will ask for
within five minutes. `key export` prompts for the passphrase. Raw key bytes in user-facing
flows are a design smell anywhere they appear; D2/D3 should make this path nearly
unreachable anyway.

### D7. Menus never offer dead ends. The Agent option routes to the real flow or doesn't exist.

**Today:** interactive `auths init` → "Agent" walks the user through capability
multi-select and a prerequisites check, *then* fails with E5007 "standalone agent
initialization is retired — run `auths id agent add`". We collect a user's choices and then
tell them their path was decommissioned. (The error text itself is good; offering the
profile is the bug.)

**The decision:** the menu entry stays — agents are a headline use case — but it *routes*:
root identity exists → run the `id agent add` delegation flow right there, reusing the
capabilities the user just selected; no root → say "an agent needs a root identity first,"
offer to create one, then continue into delegation. The user said "I want an agent."
Deliver an agent, not a history lesson about retired code paths.

### D8. Machine-consumable output goes to stdout. Period.

**Today:** `auths init --profile ci > secrets.env` captures nothing — the env block, the
single most copy-paste-destined output in the product, prints to stderr along with
everything else. Prose-to-stderr is a fine convention; the env block is not prose, it's
data. **The decision:** the env block goes to stdout (and `--json` gets a structured
equivalent). Test: every output a user would pipe must survive `>`.

### D9. The device you're on counts. `status` after init says so.

**Today:** init prints "This device authorized: did:key:zDnae…", then `status` immediately
says `Devices: none` and suggests "Link your first device → `auths pair`". To a new user
that reads as "setup didn't take" — on the success screen of the golden path. **The
decision:** status lists the current device (`this device`, with its DID) and renames the
section to other devices when relevant. Next-step hints must never contradict the line
above them.

### D10. One exit-code scheme for verify.

**Today:** artifact resolution failure exits 2; commit verification failure exits 1. CI
gates on `auths verify` — the flagship integration — get different codes for the same
semantic outcome. **The decision:** 0 = verified, 1 = verification failed, 2 = could not
attempt (I/O, malformed input). Document it in `verify --help`.

## The golden path, before → after

| Step | Today (measured) | After D1–D3 |
|------|------------------|--------------|
| Install | `brew install …` ✅ | same |
| Identity | `auths init` ✅ (~1.2s) | same |
| Sign | `git commit` *(signs but unverifiable)* | `git commit` ✅ |
| Make it verifiable | `auths sign HEAD` *(rewrites SHA!)* | — (hook did it) |
| Trust yourself | manual pin, needs unobtainable hex ❌ | — (D2/D3 defaults) |
| Verify | `auths verify HEAD` ❌ | `auths verify HEAD` ✅ |

Six user-visible steps, two of them broken → **three steps, zero of them new verbs.**
That's the whole launch story.

## What already meets the bar (don't touch it)

- **`auths demo`**: 9 ms, headless, zero prompts. The AU-1 fix holds. This is the best first-touch in the product — protect it.
- **CI profile**: 3/3 fresh-HOME inits green (could not reproduce flaky #246 here); the printed env block, applied verbatim, signs successfully. The CI handoff genuinely works.
- **`id agent add`**: clean one-shot delegation with a clear explanation of what was anchored. This is the UX bar the rest of the identity commands should match.
- **Rotation**: `id rotate` → sign works (milestone-1 stale-key fix verified). Only the verify half fails, and that's D2, not rotation.
- **Error infrastructure**: stable codes, `error list/show`, docs links. The investment shows — D5 is about picking the right code, not rebuilding the system.
- **Stateless verification**: bundle export + `--identity-bundle` verifies commits *and* artifacts. Correct design for CI; it just can't be the laptop story.

## Side notes

- **Tombstone removed error codes.** E4304 was deleted in milestone-3 cleanup, but shipped v0.1.2 binaries still emit it with a docs link. Keep anchors for removed codes in the published error docs until no released binary references them.
- **`doctor` should catch stale binaries.** This session began with a user on rc.12 against current docs — an entire debugging session caused by version drift `doctor` could detect in one comparison.
- **Wire this smoke test into CI.** ~30 s, headless, deterministic; it would have caught every finding above as a regression. Gate it once D1–D3 land (until then, B1's failures are expected-fail).
