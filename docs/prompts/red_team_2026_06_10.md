# RED_TEAM_FINDINGS — Auths (2026-06-10)

> Offensive-security engagement against the Auths decentralized-identity system.
> Recon was analysis-only; no code was modified. Every finding cites `file:line`.
> The load-bearing CRITICAL/HIGH findings (RT-001..RT-006) were independently
> re-verified against the source by the lead before publication.

---

## 1. Executive summary

**Overall posture: Critical-risk.** The cryptographic verification core — the one
component the entire product rests on — does **not** verify KEL event signatures or
inception self-certification on the stateless/embedded code path, which is exactly the
path the headline use case ("a third party verifies offline") exercises. An attacker who
can hand a verifier a Key Event Log (a CI `--identity-bundle`, a WASM widget input, an
FFI buffer, a forked Git registry) can replay a hand-crafted, unsigned KEL and obtain a
`Valid` verdict for an identity they do not control. Independently, revocation is
defeated by a signer-chosen `Auths-Anchor-Seq` trailer, agent expiry/scope is never
enforced in production (the only time-injecting verify function has no caller), and the
SCIM provisioning channel can anchor arbitrary capabilities into the org's real KEL
because its capability allowlist is dead code.

Findings by severity: **CRITICAL 3, HIGH 9, MEDIUM 9, LOW 12, INFO 3** (36 total).

The top 3 ways this system gets broken today:

1. **Forge-once on a stateless verifier (RT-001 + RT-002 + RT-005).** The embedded
   verify path replays a KEL structurally only — no event signatures, no `i == d`
   inception binding — and the `--identity-bundle` feature pins the bundle's own root.
   Craft an inception with `i = <victim or chosen prefix>` and your own key, ship the
   bundle, and `auths verify-commit` returns `Valid`. Breaks *unforgeable authorship*
   and *verifier trustworthy when embedded*.
2. **Sign forever after revocation; never expire (RT-003 + RT-004).** A revoked-but-
   unrotated key signs valid commits indefinitely by writing a low `Auths-Anchor-Seq`,
   because revocation is ordered by a value the signer controls. Agent expiry is never
   checked at all on the production path. Breaks *revocation/rotation honored* and
   *bounded delegation*.
3. **SCIM channel mints arbitrary authority (RT-006).** Every deployed SCIM tenant runs
   "permit-all" — the capability allowlist setter is never called and an empty allowlist
   means allow-everything — so any IdP channel can provision members with
   `manage_members`/deploy capabilities into the cryptographic source of truth. Breaks
   *bounded delegation*.

## 2. Attack surface map

| # | Trust boundary | What crosses it | What trusts it | How far before validation |
|---|---|---|---|---|
| B1 | `auths verify-commit --identity-bundle` (stateless CI) | attacker-supplied `IdentityBundle` JSON: root DID + full KEL + commit | the CI gate's exit code; the bundle's own root is **pushed into `pinned_roots`** (`verify_commit.rs:180-182`) | KEL replayed via structural-only `validate_kel*`; freshness-checked only (`commit_trust.rs:trusted_root_from_bundle`) |
| B2 | WASM verifier (`auths-verifier/src/wasm.rs`) | KEL JSON, attestation JSON, presentation/credential JSON from a web page | the browser widget / any JS embedder | `validate_kel` (structural), `verify_*_json`; some checks `#[cfg]`-gated weaker than native |
| B3 | C ABI verifier+keychain (`auths-verifier/src/ffi.rs`, `auths-core/src/api/ffi.rs`) | C strings, raw byte buffers + lengths, hex pubkeys | Go/Swift/Python/Node SDKs | `catch_unwind`-guarded; **one missing null-check** (RT-009); curve inferred by length at ingest |
| B4 | Control plane HTTP (`auths-api`) | JSON bodies, `Auths-Presentation` header, batch arrays | `manage_members` capability gate via `rp_auth_middleware` | presentation gate is sound; **batch-revoke unbounded** (RT-013) |
| B5 | SCIM server (`auths-scim-server`) | SCIM JSON + tenant bearer token | the org KEL (members anchored via provisioner) | tenant isolation sound; **capability allowlist dead** (RT-006) |
| B6 | Pairing daemon (`auths-pairing-daemon`) | QR/relay pair URI, ECDH pubkeys, short code, HTTP body | the device-add flow, TLS pin | rate-limiter/host-allowlist/SAS sound; cert-pin mislabeled SPKI (RT-034) |
| B7 | MCP server (`auths-mcp-server`) | JWT (RS256), KERI presentation, tool calls | downstream agent authority | JWT alg-pinned, route-gated; wildcard CORS benign (RT-035) |
| B8 | Git registry / KEL refs (`auths-storage`) | KEL events on disk under `refs/auths/*`, `refs/keri/*` | local resolution | append path runs `verify_event_crypto` (`i==d`, commitment) but **not event signatures** |
| B9 | Keychain / encrypted file (`auths-core/src/storage`) | passphrase, encrypted PKCS#8 blob | local signing | AEAD/nonce sound; **seed un-zeroized at one boundary** (RT-016); file-perm TOCTOU (RT-025) |
| B10 | Release/publish pipeline (`.github/workflows`, `scripts/releases`) | git tags, dependency graph, action refs | crates.io / npm / PyPI / Homebrew consumers | **publish not `--locked`** (RT-011); audit suppressed (RT-010); CNSA guard fictional (RT-012) |

**Reusable insight:** the verifier draws a hard line between an *append* path
(`auths-storage` → `verify_event_crypto` + commitment chaining, applied to KELs the
local device writes) and a *replay* path (`validate_kel*` → structural only, applied to
KELs that arrive over a trust boundary). The security checks live on the wrong side of
that line: the data that is most attacker-controlled (B1/B2/B3) gets the *weaker* path.

## 3. Findings register

Sorted by severity, then exploitability. IDs are stable.

### CRITICAL

#### RT-001 — Forged inception accepted: prefix never bound to its SAID on the verify path
- **Severity:** CRITICAL
- **Location:** `crates/auths-keri/src/validate.rs:641-676` (`validate_inception` adopts `icp.i` with no `i==d` check); `crates/auths-keri/src/said.rs:114-131` (`compute_said` blanks `i` for `icp`/`dip`, so the SAID is independent of `i`); the only `i==d` check lives at `crates/auths-keri/src/validate.rs:1007-1031` (`verify_event_crypto`), which is **not** called by `replay_kel_gated`/`validate_kel*`.
- **Vulnerability:** For a self-addressing AID the prefix `i` MUST equal the inception SAID `d` — that binding is what makes a KERI prefix self-certifying. `verify_event_said` only checks `d == compute_said(body)`, and `compute_said` blanks `i` before hashing, so `d` does not depend on `i`. `validate_inception` then adopts `icp.i` verbatim. Nothing on the stateless replay path asserts `i == d`.
- **Attack:** Attacker = anyone who can hand a KEL to a stateless verifier (CI `--identity-bundle` at `verify_commit.rs:173-235`; WASM `validateKelJson` at `wasm.rs:375-379`; FFI). (1) Take a victim/chosen prefix `Evictim`. (2) Craft `icp` with `i="Evictim"`, `k=[attacker_key]`, `n=[attacker_next]`; compute the real SAID over that body and set `d` to it (`d != i`). (3) Ship it; the bundle's root is pinned (RT-005). (4) `validate_kel` returns `KeyState{prefix:"Evictim", current_keys:[attacker_key]}`; the commit signed by `attacker_key` verifies → `Valid`.
- **Impact:** *Unforgeable authorship* and *verifier trustworthy when embedded* both fail. An attacker impersonates an arbitrary (and, via RT-005, a specifically pinned) identity to every stateless/embedded verifier.
- **Confidence:** Proven. `i==d` is provably absent from `validate_inception`; `compute_said` provably blanks `i`; `IcpEvent` deserializes `i != d` cleanly (`events.rs:307-316`); the one checker (`verify_event_crypto`) is provably only called on the append path (`adapter.rs:812,2090,2242`; `validate.rs:1126`) and in tests.

#### RT-002 — KEL event signatures are never verified on any verification entry point
- **Severity:** CRITICAL
- **Location:** `crates/auths-keri/src/validate.rs:486-565` (`replay_kel_gated`: SAID + sequence + chain-linkage + pre-rotation commitment only); `:858-872` (`validate_interaction` — no signature, no commitment); `:1163-1262` (`validate_signed_event` — the **only** function that verifies event signatures) has zero non-test callers; verify path consumers `crates/auths-verifier/src/commit_kel.rs:427,518,560`, `credential.rs:243`, `presentation.rs:94`, `verify.rs:215`, `wasm.rs:378` all call structural-only `validate_kel*`.
- **Vulnerability:** The verifier authorizes by replaying log *structure*. No event is ever checked to carry a valid signature from the controlling key-state on any verify path. `ixn` events have neither a commitment nor a signature gate. The code comment at `validate.rs:1633` states it outright: "`validate_kel` only checks KEL [structure]." `validate_signed_event` is grep-confirmed to be called only from tests and the signer-side multi-sig assembler (`multi_sig.rs:198`).
- **Attack:** Anyone supplying a KEL to the verifier appends a forged `ixn`/`rot`/`icp` (correct SAID + `p`-linkage, both attacker-computable, no signature). A forged `ixn` can anchor a malicious delegation seal (`Seal::KeyEvent` for an attacker `dip`) or an agent-scope/grant seal; `validate_delegation` (`validate.rs:286`) then binds the attacker delegate. Standalone, RT-001 is the cleanest single-event instance (forge an inception from scratch).
- **Impact:** *Bounded delegation*, *revocation honored*, and *unforgeable authorship* all fail. This is the **systemic root cause**: RT-001 and RT-005 largely collapse once events must be signed.
- **Confidence:** Proven (zero-caller status of `validate_signed_event`; signature-free nature of `validate_kel*` is stated in-code).

#### RT-003 — Revocation bypass via signer-asserted `Auths-Anchor-Seq`
- **Severity:** CRITICAL
- **Location:** `crates/auths-verifier/src/commit_kel.rs:318-331` (`classify_revocation`: `sign < rev` ⇒ `SignedBefore` ⇒ accepted); `:224-234` (`parse_anchor_seq` reads the trailer the signer writes); `:675-689` (`reject_unauthorized_delegate`); signer side `crates/auths-cli/src/commands/sign.rs` + `anchor_seq_trailer` at `commit_kel.rs:215-217`. Accepting path is exercised by the passing test at `tests/cases/commit_kel.rs:460-488`.
- **Vulnerability:** Revocation is ordered purely by the commit's in-band `Auths-Anchor-Seq` — a value the **signer** chooses. There is no binding tying the claimed position to the real KEL tip at signing time (no witness receipt, no transparency-log inclusion, no push-time gate). The SSH signature covers the trailer, which stops a *third party* from editing it, but the revoked signer picks the value before signing.
- **Attack:** Attacker = holder of a revoked-but-unrotated key (software keychain, key never rotated). After the org anchors the revocation at root seq `N`, the attacker crafts a commit with `Auths-Anchor-Seq: 0`, signs with the still-held key. Verifier: device KEL still validates, revocation position is `N`, `classify_revocation(Some(0), Some(N))` ⇒ `SignedBefore` ⇒ accepted ⇒ `Valid`.
- **Impact:** *Revocation honored* fails completely. Any revoked key that was not also rotated signs valid artifacts forever. (When the trailer is *absent*, the verifier fails closed — `RevokedUnknownPosition ⇒ DeviceRevoked`; the hole is precisely that a *present, forged-low* value flips it open.)
- **Confidence:** Proven (accepting path is an existing passing test; signer fully controls the trailer; no truth-binding exists).

### HIGH

#### RT-004 — Agent expiry and scope are never enforced on the production commit path
- **Severity:** HIGH
- **Location:** `crates/auths-verifier/src/commit_kel.rs:510-532` (`verify_commit_against_kel_scoped` — the only fn that passes `Some(now)`) has **zero production callers** (grep: tests + `lib.rs` re-export only); production CLI calls `verify_commit_against_kel_witnessed` (`verify_commit.rs:450`) → `authorize_commit(..., None)` (`commit_kel.rs:484`); the expiry/scope block at `:691-713` runs only when `now.is_some()`.
- **Vulnerability:** `AgentExpired` / `OutsideAgentScope` are dead verdicts on the live path. A delegator-anchored `agentscope:` seal with `expires_at` is never compared against anything in production. The org-policy path (`commit_trust.rs:evaluate_commit_policy`) can enforce expiry only if a policy was anchored at all (`NoPolicy ⇒ legacy allow`) and it explicitly contains a `NotExpired` clause (`auths-policy/src/eval.rs:236-242`, `None ⇒ allow`).
- **Attack:** Delegate an agent with `--expires`; after expiry the agent signs a commit; `auths verify` takes the `None` path; expiry is skipped; `Valid`.
- **Impact:** *Bounded delegation* fails — time-bounded delegations do not expire.
- **Confidence:** Proven (zero production callers of the scoped fn; default path passes `None`).

#### RT-005 — Stateless identity bundle pins its own root
- **Severity:** HIGH (CRITICAL in combination with RT-001/RT-002)
- **Location:** `crates/auths-cli/src/commands/verify_commit.rs:174-186` (bundle root pushed into `pinned_roots`); `:229-235` (bundle KEL used for resolution); `crates/auths-sdk/src/workflows/commit_trust.rs:trusted_root_from_bundle` (freshness check only, not a KEL/self-cert check).
- **Vulnerability:** In stateless CI the trust anchor and the evidence both come from the same untrusted file. The design is safe *only if* KERI prefixes self-certify — which RT-001/RT-002 show they do not, on this path.
- **Attack:** The delivery vehicle for RT-001: the attacker's bundle pins `did:keri:<chosen>` and supplies a matching forged/unsigned KEL; every commit "verifies." A CI gate trusting the exit code is bypassed.
- **Impact:** Defeats the CI/CD provenance gate the bundle feature exists to provide.
- **Confidence:** Proven.

#### RT-006 — SCIM capability allowlist is dead code; any tenant provisions arbitrary capabilities
- **Severity:** HIGH
- **Location:** `crates/auths-scim/src/mapping.rs:161-162` (`if allowed.is_empty() { return Ok(()) }` = permit-all); `crates/auths-scim-server/src/state.rs:77` (default `allowed_capabilities: Vec::new()`); `:90` (`with_allowed_capabilities` — grep-confirmed **no non-test caller**); `crates/auths-scim-server/src/serve.rs:9` (`TenantBootstrap` has no `allowed_capabilities` field); `crates/auths-scim-server/src/users.rs:42-45` + `provisioner.rs:175-184` (request capabilities anchored into the KEL).
- **Vulnerability:** Empty allowlist means "permit all" and the only setter is never invoked in production, so every deployed tenant runs permit-all.
- **Attack:** Attacker = holder of a SCIM bearer token (stolen, or an over-broad IdP connector). `POST /scim/v2/Users` with `auths:2.0:Agent` extension `capabilities:["manage_members","deploy:prod",...]` → anchored into the org's real KEL.
- **Impact:** *Bounded delegation* — a provisioning channel becomes an authority-granting channel writing the cryptographic source of truth.
- **Confidence:** Proven.

#### RT-007 — WASM attestation/chain verifier cannot verify compressed P-256 keys native accepts
- **Severity:** HIGH (verdict divergence; fail-closed, availability not forgery)
- **Location:** `crates/auths-crypto/src/webcrypto_provider.rs:114-144` (33-byte decompression branch is `#[cfg(feature="native")]` only) reached from `crates/auths-verifier/src/wasm.rs:190,60`; native counterpart `ring_provider.rs:94` accepts both 33- and 65-byte SEC1.
- **Vulnerability:** In a pure-WASM build, `verify_p256` errors on any 33-byte compressed pubkey — the CLAUDE.md-preferred CESR/KEL encoding. A legitimately-signed P-256 attestation that verifies on the CLI is reported invalid in the browser.
- **Attack:** No forgery; the inverse — a genuine artifact is rejected in WASM, eroding trust in the embedded verifier / pushing integrators to weaken handling.
- **Impact:** *Verifier parity* fails for the attestation/chain path on compressed P-256 (the presentation/credential path is unaffected — it uses the pure-Rust `p256` crate).
- **Confidence:** Proven (cfg-gating is explicit).

#### RT-008 — P-256 signature-malleability parity gap: native rejects high-S, WASM accepts it
- **Severity:** HIGH (verdict divergence)
- **Location:** native `crates/auths-crypto/src/ring_provider.rs:100` (RustCrypto `p256` enforces low-S) vs WASM `crates/auths-crypto/src/webcrypto_provider.rs:165-176` (`SubtleCrypto.verify` accepts both `s` and `n−s` per the WebCrypto spec).
- **Vulnerability:** For attestations routed through `CryptoProvider::verify_p256`, the two backends disagree on malleability: `(r,s)` and `(r,n−s)` are both valid in WASM, only canonical low-S natively.
- **Attack:** Anyone holding one valid P-256 attestation signature mints a second, byte-distinct signature the browser accepts but the CLI rejects — breaks signature uniqueness and creates a native/WASM verdict divergence anywhere a signature byte-string is a dedup/replay key.
- **Impact:** *Verifier parity* + signature-uniqueness. (The `software_verify` presentation path is immune — same `p256` crate on both targets.)
- **Confidence:** Likely (contingent on the deployed WebCrypto engine not normalizing, which the W3C spec does not require).

#### RT-009 — `ffi_import_key` missing null-pointer guard → undefined behavior
- **Severity:** HIGH (memory-safety, reachable from the C ABI; not remote)
- **Location:** `crates/auths-core/src/api/ffi.rs:467` (`slice::from_raw_parts(key_ptr, key_len)` with no `key_ptr.is_null()` check); sibling functions DO guard (`:856,920,967`).
- **Vulnerability:** Constructing a slice from a null base with `len>0` is immediate UB; `catch_unwind` does not catch UB.
- **Attack:** A binding bug or hostile in-process caller invokes `ffi_import_key(ctx, alias, NULL, 64, ...)` → crash or miscompile.
- **Impact:** Memory-safe-FFI claim fails for the key-import path all non-Rust SDKs call. One-line fix.
- **Confidence:** Proven.

#### RT-011 — Crates/binaries published without `--locked`: ship a dependency tree CI never tested
- **Severity:** HIGH
- **Location:** `scripts/releases/2_crates.py:123-126` (`cargo publish` no `--locked`); `.github/workflows/release.yml:75` (`cargo build --release` no `--locked`); `publish-python.yml:67` (maturin `--release` no `--locked`). No `--locked` anywhere in any build/publish path.

- **Attack:** A transitive publishes a new in-range patch between CI and the publish job; the released artifacts compile against untested code.
- **Impact:** Ship untested resolution to crates.io / GitHub Releases / Homebrew.
- **Confidence:** Proven.

#### RT-012 — Documented FIPS↔CNSA guard does not exist; CNSA silently degrades to FIPS primitives
- **Severity:** HIGH
- **Location:** `crates/auths-crypto/src/provider.rs:489-503` (comment defers to a non-existent `check-feature-sanity` xtask; the only `compile_error!` is `fips`+`wasm32` at `:508`); cfg precedence at `:531-539` lets `fips` win over `cnsa`. `crates/xtask/src/main.rs:92-104` has no `CheckFeatureSanity` variant.
- **Vulnerability:** With both features on, `fips` silently wins and the CNSA P-384 provider is excluded — a build that asked for CNSA gets FIPS primitives that **accept P-256 and ChaCha20-Poly1305** (the exact primitives CNSA forbids), with no error and no lint.
- **Attack:** A CNSA-2.0 deployment that enables `--features cnsa` in a profile that also pulls `fips` (e.g. `--all-features` in packaging) silently runs non-CNSA crypto.
- **Impact:** Silent crypto-policy downgrade; compliance violation with no detection surface.
- **Confidence:** Proven (only one `compile_error!` exists; xtask enum enumerated).

### MEDIUM

#### RT-013 — Authenticated N+1 KEL replay in batch revoke (no batch-size cap)
- **Severity:** MEDIUM
- **Location:** `crates/auths-api/src/control_plane.rs:371-385` (`batch_revoke_agents` checks only `is_empty()`); `crates/auths-id/src/keri/delegation.rs:773-792` loop → `:655-679` (`delegation_status` does a full root-KEL walk per device). No `DefaultBodyLimit` override (relies on Axum's 2 MB default).

#### RT-014 — Rotation key-deletion is best-effort and non-atomic; stale primary can survive
- **Severity:** MEDIUM
- **Location:** `crates/auths-sdk/src/domains/identity/rotation.rs:222` (`let _ = key_storage.delete_key(&old_next)`) and `:229-231` (stale-primary deletes, errors swallowed); `:167,192` (non-atomic — KEL append then keychain writes).
- **Vulnerability:** Deletes swallow errors. If `delete_key` fails (locked keychain, race), the old primary persists, re-introducing the #252/#253 nondeterministic-enumeration class on the signing side (`keychain.rs:253-268`).
- **Attack:** Local reliability/silent-correctness hazard, not remote. After a partial rotation, current-key resolution can again enumerate multiple primaries.
- **Impact:** Degrades "rotated-away key never signs again" from enforced to best-effort. Capped at MEDIUM because verification replays public keys from the KEL.
- **Confidence:** Likely (deletes are provably error-swallowing; exploit depends on keychain enumeration order).

#### RT-015 — Non-constant-time comparison of the presentation challenge nonce

- **Vulnerability:** The genuine replay-protection nonce is compared with a short-circuiting `!=`; the module uses `subtle::ConstantTimeEq` elsewhere (`auths-keri/src/crypto.rs:62-68`).

#### RT-016 — Plaintext PKCS#8 seed left un-zeroized at the storage boundary
- **Severity:** MEDIUM
- **Location:** `crates/auths-crypto/src/key_material.rs:132-147` (`build_ed25519_pkcs8_v2` returns bare `Vec<u8>`); leaked at `crates/auths-core/src/api/runtime.rs:379` and `crates/auths-core/src/crypto/ssh/keys.rs:39`. Contrast the read path which correctly returns `Zeroizing<Vec<u8>>` (`signer.rs:149-154`).

- **Attack:** Local attacker with process-memory access (core dump, swap, `/proc/<pid>/mem`) recovers the Ed25519 seed in the window after generation/rotation.
- **Impact:** Signing-seed disclosure → identity impersonation. Local only.
- **Confidence:** Proven. Fix: return `Zeroizing<Vec<u8>>`/`Pkcs8Der`.

#### RT-017 — Secure Enclave FFI passes fixed output buffers without capacity (DER signature overflow risk)
- **Severity:** MEDIUM
- **Location:** `crates/auths-core/src/storage/secure_enclave.rs:144-156,349-359,370-382` (`handle_buf=512`, `pubkey_buf=65`, `sig_buf=64`), extern decls `:20-42`.
- **Vulnerability:** Buffers are passed with `*out_len=0` but the capacity is never communicated to the Swift side, which must hard-code matching constants. A DER-encoded ECDSA signature (~72 bytes) written into the 64-byte `sig_buf` overflows the Rust-owned heap before the length is read; the post-call `truncate` can only shrink.
- **Attack:** Build-integration footgun, not remote — triggers if the Swift `se_sign` returns DER rather than raw `r||s`.
- **Impact:** Potential heap overflow in the on-device signing path.
- **Confidence:** Likely (missing capacity arg is proven; overflow depends on the out-of-tree Swift output encoding). Confirm the Swift `se_sign` encoding + add a capacity-in argument.

#### RT-018 — Byte-length curve dispatch at ingest boundaries (banned anti-pattern, latent)
- **Severity:** MEDIUM (latent silent-correctness; LOW today)
- **Location:** `crates/auths-verifier/src/ffi.rs:18-25` (`pk_from_bytes_ffi`), `crates/auths-verifier/src/wasm.rs:18-32` (`pk_from_hex_wasm`), `crates/auths-verifier/src/core.rs:678-696` (legacy bare-hex `DevicePublicKey` deserialize), `packages/auths-verifier-swift/src/lib.rs:21-35`, and the Python/Go hex helpers — all map `32→Ed25519, 33|65→P-256`.
- **Vulnerability:** Directly violates the CLAUDE.md "never dispatch on pubkey byte length" rule. `xtask/check_curve_agnostic.rs:16` bans `from_public_key_len_fallback` but not these length-match arms.
- **Attack:** None today (32/33/65 are disjoint). The moment a 32-byte (X25519) or 33-byte (secp256k1, already a feature flag) curve lands, a key routes to the wrong algorithm and fails as `InvalidSignature` instead of a routing error.
- **Impact:** Latent verifier-parity / silent-correctness hazard in the embeddable verifier. (The KERI credential/presentation path is unaffected — it dispatches on the in-band CESR tag.)
- **Confidence:** Proven (length-match in source); exploitability Hypothesis.

### Milestone 1 — CRITICAL (Epic A: Verifier fail-closed hardening)

#### TASK A.0 — Make the verify path signature-typed (retire the bug-class)
**Severity:** CRITICAL (structural prevention — this is what stops RT-001/002/005 *recurring*)
**Fixes:** RT-002 class at the root (the durable counterpart to A.1's logic)
**Vulnerability:** The bug-class behind all three CRITICALs is that a *structural-only* replay (`validate_kel`/`replay_kel_gated`) is **public** and is the function every untrusted-input entrypoint reaches for. As long as a structural-only API exists and is callable with bare `Event`s, the next new entrypoint (a future language port, a new FFI export) can forget the signature/`i==d` check and silently reintroduce the forge. A test catches an instance; only the type system retires the class.
**Files:** `crates/auths-keri/src/validate.rs` (signatures of `validate_kel`/`validate_kel_with_lookup`/`validate_kel_with_receipts`/`replay_kel_gated`), `crates/auths-keri/src/events.rs` (`SignedEvent`), `crates/auths-verifier/src/{commit_kel,credential,presentation,verify,wasm,ffi}.rs`, `crates/auths-verifier/src/core.rs` (`IdentityBundle` wire), `crates/xtask/src/` (new `check-verify-path-completeness`).
**Fix:** Make it *unrepresentable* to replay unverified events. (1) Change the public replay API to accept only `&[SignedEvent]` (events carrying their CESR attachment); the bare-`Event` structural replay becomes `pub(crate)`/private, reachable only through the signed wrapper. (2) Update every wire format (`IdentityBundle.kel`, WASM/FFI inputs) to carry signatures so they deserialize into `SignedEvent`. (3) Add an xtask `check-verify-path-completeness` that fails the build if the structural-only replay has any caller outside the signed wrapper module. A.1 then fills the verification logic *inside* this typed boundary.
**Acceptance:** No public function replays bare `Event`s; calling `validate_kel` with unsigned events is a **compile error**; `check-verify-path-completeness` is green and merge-blocking. (A.0 is the scaffold, A.1 the logic — both must land together to be meaningful.)
**Effort:** L (mostly mechanical signature/wire-format churn; pre-launch, no compat burden)
**Regression risk:** Medium — touches every verify entrypoint's signature. De-risk: land A.0+A.1 on one branch behind the M0 corpus; happy-path fixtures must stay green.
**Depends on:** 0.1

#### TASK A.1 — Verify KEL event signatures on every replay path
**Severity:** CRITICAL
**Fixes:** RT-002 (and largely collapses RT-001/RT-005)
**Vulnerability:** `validate_kel*`/`replay_kel_gated` authorize by log structure alone; no event is checked to be signed by the controlling key-state. `validate_signed_event` (the only signature-verifying function) has zero non-test callers. An attacker supplies a KEL with forged `icp`/`ixn`/`rot` events and the verifier replays them as authoritative.
**Files:** `crates/auths-keri/src/validate.rs` (`replay_kel_gated`, `validate_kel`, `validate_kel_with_lookup`, `validate_kel_with_receipts`), `crates/auths-verifier/src/commit_kel.rs`, `credential.rs`, `presentation.rs`, `wasm.rs`, `ffi.rs`; bundle/KEL wire structs must carry detached signatures.
**Fix:** Inside the `SignedEvent`-typed `replay_kel_gated` from A.0, verify each event against the in-force key-state during replay (fold `validate_signed_event`'s logic in). Inception/dip must verify the signature against `k[]` under `kt`; `rot`/`drt` against the new keys with the prior-commitment binding; `ixn` against the current key-state.
**Acceptance:** TASK 0.1 (b)/(c) pass; an unsigned or wrong-signer event yields a fail-closed verdict in native, WASM, and FFI. Add a CI test asserting `validate_signed_event`-equivalent coverage on every public verify entry.
**Effort:** XL (break into: replay verifies icp/dip; replay verifies rot/drt; replay verifies ixn — the wire-format/typing churn is A.0).
**Regression risk:** High — this is the core path. De-risk with the M0 corpus + the existing happy-path fixtures (must stay green) and a staged rollout per event type.
**Depends on:** 0.1, A.0

> **Implementation sketch (top CRITICAL #1).** The cleanest design makes the replay
> path operate on `SignedEvent` rather than `Event`. (1) A.0 has already changed
> `validate_kel*` to take `&[SignedEvent]` and made `IdentityBundle.kel` / WASM / FFI
> inputs carry the CESR attachment. (2) Inside `replay_kel_gated`, after `verify_event_said`
> and *before* mutating `state`, call the per-event signature check against the
> key-state that authorizes that event (inception/dip against its own `k`/`kt`; rot/drt
> against new keys + prior commitment; ixn against `state.current_key`). (3) Keep
> structural checks where they are. **Trap to avoid:** do not verify the rot signature
> against the *new* keys alone — KERI binds a rotation to the *prior* commitment (already
> checked) AND a signature by the keys being rotated in; getting this wrong either breaks
> rotation or re-opens the forge. Mirror `validate_signed_event` exactly; it already
> encodes the dual-index logic.

#### TASK A.2 — Enforce inception self-certification (`i == d`) on the replay path

**Vulnerability:** `validate_inception` adopts `icp.i` as the controller prefix without asserting `i == d`; `compute_said` blanks `i`, so the SAID does not bind the prefix. A forged inception claims any prefix with attacker keys.

> **Implementation sketch (top CRITICAL #2).** Add, at the top of `validate_inception`,

#### TASK A.3 — Bind the signing position to a verifiable anchor (kill self-asserted anchor-seq)
**Severity:** CRITICAL
**Fixes:** RT-003
**Vulnerability:** Revocation ordering trusts `Auths-Anchor-Seq`, a trailer the signer writes. A revoked-but-unrotated key signs `Valid` commits forever by claiming a pre-revocation position.
**Files:** `crates/auths-verifier/src/commit_kel.rs` (`classify_revocation`, `parse_anchor_seq`, `reject_unauthorized_delegate`), `crates/auths-cli/src/commands/sign.rs`, `crates/auths-sdk/src/domains/identity/local.rs`.
**Fix:** Do not trust an in-band, signer-chosen position for revocation ordering. Options, in order of strength: (a) bind the commit to a witness receipt / transparency-log inclusion proof of the device's KEL state at signing time and order revocation against *that*; (b) default to fail-closed — a commit by a *currently-revoked* delegate is `DeviceRevoked` regardless of any claimed prior position, accepting only commits whose authorizing KEL state can be independently shown to predate the revocation (e.g. the commit is reachable in history before the revocation anchor). Given pre-launch posture, (b) is the correct minimal fix: treat revocation as terminal for new signatures and stop honoring a self-reported "I signed earlier."
**Acceptance:** TASK 0.1 (d) passes; a revoked signer cannot produce a `Valid` commit by lowering the trailer. Add a CI test that a forged-low anchor-seq is rejected.
**Effort:** L
**Regression risk:** Medium — must not retroactively invalidate genuinely-prior history. De-risk by ordering against an independently verifiable fact (witness/transparency or commit-graph reachability), not the trailer.
**Depends on:** 0.1

> **Implementation sketch (top CRITICAL #3).** The bug is that the trust direction is
> inverted — the verifier asks the revoked party where it stands. Replace
> `classify_revocation`'s reliance on `parse_anchor_seq` with an external ordering
> source. Minimal correct version: if the delegate is revoked at root seq `N`, reject any
> commit whose authorizing delegation cannot be *independently* placed before `N` — and
> "independently" must not be the signer's own trailer. The simplest independent signal
> available today is the witness-receipted KSN (already scaffolded in
> `auths-keri/src/witness/`) or, absent witnesses, the commit's position in the signed
> Git history relative to the revocation anchor. **Trap to avoid:** do not simply delete
> the `SignedBefore` branch — that would invalidate legitimate historical commits;
> replace its *input* with a non-attacker-controlled one.

#### TASK A.4 — Stop self-pinning bundle roots without independent self-certification

## 5. Adversarial test gaps

The negative/abuse tests this codebase should have but doesn't (these are M0):

- **Forged inception** — `icp` with `i != d`; basic-derivation `i != k[0]` → must reject. (RT-001)
- **Unsigned/forged events** — `ixn` anchoring a delegation/scope seal with no signature; `rot` whose new key is not signed by the prior key → must reject. (RT-002)
- **Revoked-key forged position** — revoked-but-unrotated signer with `Auths-Anchor-Seq` below the revocation position → must be `DeviceRevoked`. (RT-003)
- **Expired agent** — delegated signer past its anchored `expires_at` through the *production* verify entry → must be `AgentExpired`. (RT-004)
- **Self-pinning bundle** — bundle whose root does not equal `compute_said(kel[0])` → must fail closed. (RT-005)
- **SCIM over-provision** — tenant requests `manage_members`/deploy outside its allowlist (create AND patch) → must reject. (RT-006)
- **Native↔WASM differential** — compressed P-256 key, high-S signature, wrong-curve key, duplicate JSON keys, oversized input, trailing data → identical verdict across Rust/WASM/Go/Swift/Python/TS. (RT-007, RT-008, RT-018, RT-032)
- **FFI fuzz/abuse** — null pointers, zero/huge lengths, non-UTF-8, DER vs raw SE signatures → no crash/UB. (RT-009, RT-017)
- **Batch DoS bound** — oversized batch-revoke array → 400, linear cost. (RT-013)
- **Fuzz targets** — `validate_kel`, `verify_*_json`, CESR attachment parser. (RT-001/002 class)

## 6. Accepted-risk reconciliation

The repo documents accepted risks in `docs/architecture/multi_device_accepted_risks.md`
(`kt=1` duplicity, no witnesses) and `docs/architecture/pin-kill-switch-threat-model.md`.

| Finding | Verdict | Rationale |
|---|---|---|
| Duplicity / forked KEL (`detect_duplicity`) | **Within envelope** | The detector fires on same-`(prefix,seq)` differing-SAID and is surfaced as a non-fatal warning; this matches the documented `kt=1` trust-on-first-sight model. No drift. |
| Pair-URI size bound (`SHARED_KEL_INCEPTION_EVENT_MAX_BYTES`) | **Within envelope** | Enforced in `SubmitResponseRequest::validate()` (`auths-pairing-protocol/src/types.rs:158-169`); holds against oversized multi-sig inceptions. |
| TLS pin fail-closed / cert-pin labeling (RT-034) | **Within envelope** | The pin-kill-switch threat model accepts fail-closed pinning; the SPKI-vs-cert naming is a hardening note, not a model violation. |
| **RT-003 revocation bypass** | **EXCEEDS envelope** | The accepted-risk doc assumes the anchor-seq reflects reality. Reality is that it is attacker-chosen. Revocation defeat is NOT a documented accepted risk. |
| **RT-001/RT-002 missing self-cert & signatures** | **EXCEEDS envelope** | The docs assume KERI prefixes self-certify and events are authenticated; the stateless path does neither. This is a verification-correctness break, distinct from the `kt=1`/witness gaps. |
| **RT-004 unenforced expiry** | **EXCEEDS envelope** | Time-bounded delegation is a stated capability; it is silently unenforced in production. |

The KERI wire-format divergences (in-body `dt`, `1AAI` P-256 code, cleared `d`/`i`
signing) catalogued in `docs/plans/keri_compliance.md` are **within** the documented
Epic-4 envelope and were not re-litigated here — except where they intersect a real
finding: A.1/A.2 should be designed alongside Epic 4's "sign over finalized bytes" (F-06)
and "seal `Said`/`Prefix` types" (F-10/T-01) so the fixes don't fight each other.

## 7. Open questions for the human

1. **Stateless-bundle trust model.** Is `auths verify-commit --identity-bundle` meant to
   trust a root *supplied by the artifact under verification*, or only to provide
   *evidence for* an independently-pinned root? RT-005's fix depends on the answer. (My
   recommendation: bundle is evidence, never the pin source.)
2. **Revocation ordering source.** With no witnesses today (accepted risk), what
   independent signal should order a commit against a revocation — witness receipts (Epic
   3), a transparency log, or Git-history reachability? RT-003's strongest fix needs this
   decision; the interim fail-closed fix (treat revocation as terminal for new signatures)
   works without it.
3. **Is expiry/scope enforcement intended for launch?** RT-004 shows the only verify
   function that enforces it is unreachable. Either wire it into the production path or
   explicitly mark time-bounded delegation as post-launch.
4. **SCIM default posture.** Should an unconfigured tenant allowlist deny all (my
   recommendation) or is permit-all an intended convenience for single-tenant pilots?
   B.1 flips a default.
5. **CNSA deployment reality.** Is anyone actually building `--features cnsa`? If CNSA is
   not a launch target, E.2 can simply delete the dead provider + comment rather than add
   the guard.
6. **Multi-tenant key storage.** CLAUDE.md points toward agents/servers reusing the SDK
   in-process. Is multi-identity-per-process a launch scenario? RT-024's severity rises
   sharply if so.

## 8. CI enforcement matrix — keep these from recurring

The deepest lesson of this audit is not any single `RT-NNN` — it is the **pattern**:
security checks lived on the *append* path but not the *replay* path, and the discipline
that would have caught it (negative tests, a verify-path lint, a differential harness)
did not exist. The per-task `Acceptance` lines above each name a regression test; this
section consolidates them into one strategy so a reviewer can see the gate at a glance.

**Two principles:**

1. **Prefer making the bug unrepresentable over catching it.** Strength order:
   **type-level (compile error) > xtask static lint > adversarial test > fuzz.** A type
   retires a class; a test only catches one instance of it. TASK A.0 is the embodiment —
   it is more valuable than any test in this report.
2. **Every security-relevant check must be merge-blocking.** Prior art in this repo:
   `check-arch.sh` was failing *and ungated* for a period (it is now gated at
   `ci.yml:81`). An advisory-only check decays. If a gate below is not wired to fail the
   build, it does not count.

| Layer (strongest first) | Mechanism | Guards | Owning task | Gating |
|---|---|---|---|---|
| **Type-level** | Replay API accepts only `&[SignedEvent]`; bare-`Event` replay is `pub(crate)`/private | RT-001, RT-002, RT-005 | A.0 | compile error |
| **xtask static** | `check-verify-path-completeness` — structural replay has no caller outside the signed wrapper | RT-002 class (new entrypoints) | A.0 | **block** |
| **xtask static** | extend `check_curve_agnostic` to ban `match ….len() { 32 \| 33 \| 65 => }` curve dispatch | RT-018 | C.2 | **block** |
| **xtask static** | `check-feature-sanity` — `compile_error!` on `fips`+`cnsa`; assert `audit.toml`/`deny.toml` ignore-lists match | RT-010, RT-012, RT-019 | E.2, E.3 | **block** |
| **xtask static** | lint banning error-swallowing (`let _ = …delete_key(`/security-relevant `let _ =`) | RT-014 | F.3 | **block** |
| **Adversarial tests** | M0 negative corpus: forged `icp` (`i!=d`), unsigned `ixn`/`rot`, revoked-low-anchor-seq, expired agent, self-pin bundle, SCIM over-provision (create+patch) | RT-001/002/003/004/005/006 | 0.1, A.2, A.3, A.4, B.1 | **block** |
| **Differential** | same vector set through native / WASM / Go / Swift / Python / TS → identical verdict (compressed P-256, high-S, wrong-curve, duplicate JSON keys, oversized, trailing data) | RT-007, RT-008, RT-018, RT-032 | 0.2, C.3, C.4 | **block** |
| **Fuzz** | `cargo fuzz` on `validate_kel` / `verify_*_json` / CESR attachment parser, seeded from the M0 corpus; FFI abuse (null/huge-len/non-UTF-8) | RT-001/002 class, RT-009, RT-017 | 0.3 | smoke (time-boxed) in CI; nightly deep run |
| **Workflow** | `--locked` on all build/publish; SHA-pin third-party actions; `--frozen-lockfile`; publish gated on the commit-verify job | RT-011, RT-020, RT-021, RT-029, RT-030 | E.1, E.4, F.4 | **block** |

**The single most important addition** is the type-level guard (A.0) plus its
`check-verify-path-completeness` lint. Everything else catches a *known* finding
regressing; A.0 prevents an *unknown future* entrypoint from reintroducing the entire
class. Without it, the verify-path asymmetry that produced RT-001/002 will return the
next time someone adds a verifier surface.

**Suggested CI wiring:** fold the static lints into the existing `xtask-checks` job
(`ci.yml:81`, already gated); add a `verifier-conformance` job that runs the M0 corpus +
the cross-language differential and is a required status check on `main`; add a
`fuzz-smoke` step (time-boxed) to PR CI and a scheduled deep-fuzz workflow.
