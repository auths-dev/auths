What’s being compared

Every identity an organization creates — an employee, a device, an AI agent — is vouched for by an entry in the organization’s shared, tamper-evident log. Anyone verifying an identity replays that log. The two implementations differ only in how those entries are written.

Per-identity anchoring · today

Each new identity gets its own log entries — three of them — and each is saved to disk separately. Think of a notary who writes, stamps, and files a fresh page for every single hire. It’s simple, but the book grows three pages per identity, and adding each new page gets slower as the book gets thicker.

Batch anchoring · proposed

New identities are gathered into a group, and the notary stamps one page listing the whole cohort, filed once. The book stays short no matter how many identities join — so writing stays fast and reading stays fast. Each individual is still verified exactly as before, against that one shared page.

Same cryptography, same verification, same revocation — only the bookkeeping differs. Every run below passes identical verification checks on both implementations.
