# fn-12.14 Set up automated publish pipeline

## Description

Set up `cargo publish --workspace` and optionally integrate `release-plz` for automated CI-based publishing.

### Steps

1. **Verify `cargo publish --workspace --dry-run` passes**
   - All workspace deps must have `path` + `version` (already done in root `Cargo.toml`)
   - No circular dependencies (resolved by Phase 1-2)
   - `xtask` must have `publish = false`
   - Mark any internal-only crates as `publish = false`

2. **Define explicit publish order** in documentation:
   ```
   Tier 0: auths-crypto, auths-policy, auths-telemetry, auths-index
   Tier 1: auths-verifier, auths-core (verifier before core)
   Tier 2: auths-id
   Tier 3: auths-storage, auths-infra-git, auths-infra-http
   Tier 4: auths-sdk
   Tier 5: auths-cli
   ```
   (Adjusted based on fn-12.13 decisions)

3. **Consider `release-plz` integration**:
   - Create `release-plz.toml` with workspace config
   - Create `.github/workflows/release-plz.yml`
   - Configure `semver_check = true` for public crates
   - Configure `publish = false` for internal crates

4. **Add CI step for `--all-features` testing** to catch `test-utils` regressions

### Files to create/modify

- `release-plz.toml` (new)
- `.github/workflows/release-plz.yml` (new, optional)
- `.github/workflows/ci.yml` — add `--all-features` test step
- `crates/xtask/Cargo.toml` — ensure `publish = false`

### Verification

```bash
cargo publish --workspace --dry-run
```
