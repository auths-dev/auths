## Harden zeroization at FFI boundaries

### Context

Private key material currently crosses the FFI boundary as hex-encoded `String`:
- `IdentityResult.current_key_pkcs8_hex: String` (line 308 of auths-mobile-ffi/src/lib.rs)
- `PairingResult.shared_secret_hex: String` (ECDH shared secret, line 707)

Once hex-encoded into a `String` and passed to Swift/Kotlin, the Rust side cannot zeroize it. The host language GC controls when the memory is reclaimed. This leaves sensitive key material in memory indefinitely.

### What to Do

1. Replace hex string returns with opaque key handles:
   ```rust
   #[derive(uniffi::Record)]
   pub struct KeyHandle {
       id: u64,
   }
   ```

2. Implement a Rust-side key store (in-process, not persistent):
   - `HashMap<u64, Zeroizing<Vec<u8>>>` behind a `Mutex`
   - Handles are monotonically increasing IDs
   - Keys are stored as `Zeroizing<Vec<u8>>` — zeroized when dropped
   - Provide `release_key(handle)` to explicitly drop + zeroize
   - Consider TTL-based cleanup for leaked handles

3. API changes:
   - `create_identity()` → returns `KeyHandle` instead of `current_key_pkcs8_hex: String`
   - `sign_with_identity()` → accepts `KeyHandle` instead of hex key string
   - `create_pairing_response()` → returns `KeyHandle` for shared secret
   - New: `export_key(handle) -> Vec<u8>` for when the app explicitly needs raw bytes (e.g., to store in native keychain) — document that this is the point of no return for zeroization

4. Add `drop_all_keys()` for app lifecycle events (backgrounding, logout).

### Key Files

- `crates/auths-mobile-ffi/src/lib.rs:308` — current_key_pkcs8_hex String
- `crates/auths-mobile-ffi/src/lib.rs:706-707` — shared_secret_hex String
- `crates/auths-core/src/signing.rs` — uses `Zeroizing<String>` internally

### Constraints

- `KeyHandle` must be `Send + Sync` (just a u64, trivially so)
- Key store must be thread-safe (`Mutex<HashMap>`)
- `Zeroizing<Vec<u8>>` for all stored key material
- `export_key()` must document the security trade-off clearly
- Handle-based API must be ergonomic for mobile developers
