## Description
# Task: Harden KERI Event Serialization Field Order

Currently, KERI event structs (`IcpEvent`, `RotEvent`, `IxnEvent`) use derived `Serialize` impls, which serialize fields in Rust struct declaration order. Reordering struct fields silently breaks every existing SAID. This task makes the serialization order explicit and tested.

## Why this matters

KERI interoperability requires matching the field order used by the ecosystem (keripy, keriox). The spec-defined order is: `v, t, d, i, s, kt, k, nt, n, bt, b, a, p, x`. If a future refactor reorders struct fields, SAIDs break silently â€” no compiler error, no test failure (until now).

## What to do

### 1. Add custom `Serialize` impls to KERI event structs

Replace `#[derive(Serialize)]` with hand-written `impl Serialize` that enforces the KERI spec field order regardless of Rust struct layout.

Affected structs (two parallel sets â€” both must be hardened):

**auths-verifier** (Layer 1, canonical definitions):
- `IcpEvent` at `auths-verifier/src/keri.rs:353-385`
- `RotEvent` at `auths-verifier/src/keri.rs:389-423`
- `IxnEvent` at `auths-verifier/src/keri.rs:427-444`

**auths-id** (Layer 3, richer types):
- `IcpEvent` at `auths-id/src/keri/event.rs:87-115`
- `RotEvent` at `auths-id/src/keri/event.rs:124-154`
- `IxnEvent` at `auths-id/src/keri/event.rs:163-180`

The custom impl pattern:
```rust
impl Serialize for IcpEvent {
    fn serialize<S: Serializer>(&self, serializer: S) -> Result<S::Ok, S::Error> {
        use serde::ser::SerializeMap;
        let mut map = serializer.serialize_map(None)?;
        map.serialize_entry("v", &self.v)?;
        map.serialize_entry("t", "icp")?;
        map.serialize_entry("d", &self.d)?;
        map.serialize_entry("i", &self.i)?;
        map.serialize_entry("s", &self.s)?;
        // ... remaining fields in spec order
        map.end()
    }
}
```

### 2. Add pinning tests for serialized byte output

For each event type, serialize a known event with known field values and assert the exact JSON output bytes. This catches any accidental reordering â€” in custom impls, in serde behavior changes, or in upstream dependency updates.

Place tests in:
- `auths-verifier/tests/cases/serialization_pinning.rs`
- `auths-id/tests/cases/serialization_pinning.rs`

### 3. Add cross-validation between auths-verifier and auths-id

Extend the existing SAID cross-validation pattern at `auths-core/tests/cases/said_cross_validation.rs` to also verify that both event type systems produce identical serialized bytes for equivalent events.

### 4. Keep `Deserialize` derived

Only `Serialize` needs hardening. `Deserialize` is field-name-based (not order-dependent) so `#[derive(Deserialize)]` remains correct and safe.
