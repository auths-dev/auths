## Add SyncCryptoProvider trait to auths-crypto

### Context

`CryptoProvider` at `crates/auths-crypto/src/provider.rs:85-138` is an `async_trait` because `WebCryptoProvider` (WASM) requires async. On native targets, `RingCryptoProvider` wraps synchronous `ring` calls in async — pure overhead. The FFI path needs sync crypto to avoid pulling in a tokio runtime.

### What to Do

1. Add a `SyncCryptoProvider` trait in `crates/auths-crypto/src/provider.rs` mirroring `CryptoProvider` but with synchronous method signatures (no async, no Future).

2. Implement `SyncCryptoProvider` for `RingCryptoProvider` — this should be trivial since `ring` calls are already sync. The existing async impl just wraps them.

3. Gate `SyncCryptoProvider` behind a feature flag (e.g., `sync` or include it unconditionally since it has no extra deps).

4. Keep `CryptoProvider` (async) unchanged — WASM still needs it.

5. Update `crates/auths-core/src/crypto/provider_bridge.rs` to optionally use `SyncCryptoProvider` directly when available, bypassing the `FALLBACK_RT` / `block_in_place` pattern.

### Key Files

- `crates/auths-crypto/src/provider.rs:85-138` — CryptoProvider trait (async_trait)
- `crates/auths-crypto/src/ring_provider.rs` or wherever RingCryptoProvider lives — sync ring calls wrapped in async
- `crates/auths-core/src/crypto/provider_bridge.rs:1-55` — existing sync bridge with FALLBACK_RT
- `crates/auths-crypto/Cargo.toml` — tokio is optional, gated behind `native` feature

### Constraints

- Do NOT remove or modify the async `CryptoProvider` — WASM depends on it
- `SyncCryptoProvider` methods should have identical signatures minus `async`
- All existing tests must continue passing
- Follow the project's trait pattern: `&self` methods, `Send + Sync` bounds
