## Triage and fix auths-cli production unwraps

The CLI crate has ~123 unwrap/expect calls. Since auths-sign and auths-verify run as git hooks, panics here directly kill the user's commit flow — this is the actual risk surface described in the epic justification.

### Steps
1. Run `cargo clippy -p auths-cli --all-targets -- -W clippy::unwrap_used -W clippy::expect_used 2>&1` to get the full list
2. Triage each occurrence:
   - **Clap argument parsing**: Typically safe after clap validation — `#[allow]` with justification
   - **Config file parsing**: Should return proper errors to user — convert to `?`
   - **Git operations**: Should return proper errors — convert to `?`
   - **String formatting/display**: Usually infallible — `#[allow]` with justification
   - **Test-only**: Should be covered by `allow-unwrap-in-tests` already
3. For the CLI as a presentation layer, some `expect()` calls are acceptable (e.g., "failed to write to stdout") with `#[allow]` annotations
4. Focus on the `auths-sign` and `auths-verify` binary paths first — these are the git hook hot paths

### Decision: CLI is presentation layer
Per CLAUDE.md, the CLI is the translation boundary. Some panics (e.g., stdout write failure) are acceptable with annotation. The goal is to ensure no user-input-dependent panics, not zero panics.
