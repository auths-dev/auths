## Implement batch_append_events on GitRegistryBackend with state overlay

Add `batch_append_events` as an inherent method on `GitRegistryBackend` (NOT on the frozen `RegistryBackend` trait).

### Method signature

```rust
impl GitRegistryBackend {
    /// Applies multiple KEL events in a single Git commit.
    ///
    /// Args:
    /// * `events`: Ordered list of (prefix, event) pairs to apply atomically.
    ///   If any event fails validation, the entire batch is rejected.
    ///
    /// Usage:
    /// ```ignore
    /// backend.batch_append_events(&events)?;
    /// ```
    pub fn batch_append_events(
        &self,
        events: &[(Prefix, Event)],
    ) -> Result<(), RegistryError>
}
```

Where `Prefix` = `auths_verifier::keri::Prefix` and `Event` = `auths_id::keri::event::Event`.

### Implementation

1. Acquire `AdvisoryLock` (POSIX flock on `registry.lock` â€” held for entire batch)
2. Read tip commit ref
3. Create `TreeMutator::new()`
4. Create `HashMap<Prefix, KeyState>` as in-memory state overlay
5. For each `(prefix, event)` at index `i`:
   a. Look up current state: first check overlay, then read from Git tree
   b. Validate event against current state (same validation as `append_event`)
   c. Stage 3 blob writes via `TreeMutator::write_blob()`:
      - `{prefix_path}/events/{seq}.json` â€” the raw event
      - `{prefix_path}/tip.json` â€” latest SAID + sequence
      - `{prefix_path}/state.json` â€” computed `CachedStateJson`
   d. Compute next `KeyState` via:
      - ICP â†’ initial state construction
      - ROT â†’ `KeyState::apply_rotation()`
      - IXN â†’ `KeyState::apply_interaction()`
   e. Update overlay with new `KeyState`
   f. Track metadata counter changes in-memory
6. Write metadata updates (`identity_count`, `device_count`) as final blobs
7. `TreeMutator::build_tree()` with parent tree as base
8. Create commit with CAS (verify tip ref hasn't changed since step 2)
9. Lock released on drop

### Error handling

Add new error variant to `RegistryError`:
```rust
BatchValidationFailed { index: usize, source: Box<RegistryError> }
```

Map validation failures at step 5b to `BatchValidationFailed { index: i, source }`.

Empty batch: return `Ok(())` (no-op, no commit created).

## Implement batch_append_events on GitRegistryBackend with state overlay

Add `batch_append_events` as an inherent method on `GitRegistryBackend` (NOT on the frozen `RegistryBackend` trait).

### Method signature

```rust
impl GitRegistryBackend {
    /// Applies multiple KEL events in a single Git commit.
    ///
    /// Args:
    /// * `events`: Ordered list of (prefix, event) pairs to apply atomically.
    ///   If any event fails validation, the entire batch is rejected.
    ///
    /// Usage:
    /// ```ignore
    /// backend.batch_append_events(&events)?;
    /// ```
    pub fn batch_append_events(
        &self,
        events: &[(KeriPrefix, Event)],
    ) -> Result<(), RegistryError>
}
```

### Implementation

1. Acquire advisory lock (held for entire batch)
2. Read tip commit
3. Create `TreeMutator` from parent tree
4. Create `HashMap<KeriPrefix, IdentityState>` as in-memory state overlay
5. For each `(prefix, event)`:
   a. Look up current state: first check overlay, then read from tree
   b. Validate event against current state
   c. Stage event mutations via TreeMutator
   d. Update overlay with new state
   e. Update metadata counters in-memory
6. Build final tree, create single commit
7. Release lock

### Error handling

Add new error variant:
```rust
BatchValidationFailed { index: usize, source: Box<RegistryError> }
```

Empty batch: return `Ok(())` (no-op, no commit created).

### State overlay

The overlay prevents re-reading Git for sequential events on the same identity. When a batch contains ICP at seq 0 then IXN at seq 1 for the same prefix, the IXN validation uses the state computed from the ICP, not the pre-batch Git state.

### Metadata

`identity_count` and `device_count` must be tracked in-memory during the batch and written once at the end. Don't re-read metadata from Git after each event.
