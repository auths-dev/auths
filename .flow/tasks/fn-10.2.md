## Refactor CLI publish.rs to use SDK publish_artifact workflow

### Context

After fn-10.1, the SDK has `publish_artifact()`. Now thin out the CLI handler.

### What to Do

**1. Update `crates/auths-cli/src/commands/artifact/publish.rs`:**

- Remove `transmit_publish()` function entirely
- Remove `validate_package_identifier()` function (now in SDK)
- Remove the ad-hoc `reqwest::Client` construction at line 165
- In `handle_publish_async()`: read the attestation file, call `sdk::publish_artifact(&registry_client, ...)`, match on `PublishOutcome` / `PublishError` for user-facing output
- The `registry_client` should come from `HttpRegistryClient` (already available via `auths-infra-http` dependency)
- Keep all `println!`, `eprintln!`, `console::style`, `is_json_mode()` formatting in the CLI

**2. Wire up `HttpRegistryClient` in CLI composition root if needed:**

- Check if `crates/auths-cli/src/factories/mod.rs` already creates an `HttpRegistryClient`
- If not, create one from the shared `reqwest::Client` in `CliConfig`
- Pass it to the publish handler

**3. Handle the sync/async boundary:**

- The current `handle_publish()` creates its own `tokio::Runtime`. This pattern stays in the CLI -- the SDK function is async, the CLI wraps it.
