# fn-12.13 Evaluate auths-core to auths-verifier dependency

## Description

Evaluate the `auths-core` dependency on `auths-verifier` and determine the best resolution. Currently `auths-core` (intended Layer 0) depends on `auths-verifier` (Layer 1) at `Cargo.toml:53`, which breaks the layered architecture.

### Current usage

`auths-core` imports from `auths-verifier` in 30+ sites, using types like:
- `ClockProvider`, `Prefix`, `Said`, `DeviceDID`, `Ed25519PublicKey`, `Attestation`

### Options to evaluate

1. **Make `auths-verifier` optional** — requires `#[cfg(feature)]` gating across 30+ files. Likely too invasive.

2. **Extract shared types to a new Layer 0 crate** (e.g., `auths-types`) — `Prefix`, `Said`, `DeviceDID`, `Attestation`, `ClockProvider` move there. Both `auths-core` and `auths-verifier` depend on `auths-types`.

3. **Accept `auths-core` as Layer 1** — the simplest option. Adjust the architecture diagram. `auths-core` stays Layer 1 alongside `auths-verifier`. The publish order adjusts to: verifier before core.

4. **Move the shared types into `auths-crypto`** — since `auths-crypto` is already Layer 0 and both depend on it.

### Deliverable

This task produces a **decision document** and, if option 2 or 4, the implementation. If option 3, update the architecture diagram in the epic spec.

### Also evaluate: `auths-infra-git` -> `auths-sdk` dependency

- Move `GitLogProvider` trait down to `auths-core::ports::git`
