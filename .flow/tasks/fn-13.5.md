# fn-13.5 Eliminate unwrap/expect in core and SDK production code

## Description
Audit and eliminate `unwrap()`/`expect()` from `auths-core` and `auths-sdk` production code. Add clippy lint enforcement to prevent regressions.

### What to do

1. **Audit targets in `auths-core`** (production code only):

   **Mutex sites in `signing.rs`** — 5 sites, all in `CachedPassphraseProvider` and `UnifiedPassphraseProvider`:
   - L386 `self.cache.lock().unwrap().clear()` in `clear_cache()` → returns `()`
   - L392 `self.cache.lock().unwrap()` in `get_passphrase()` → returns `Result<Zeroizing<String>, AgentError>`
   - L409 `self.cache.lock().unwrap()` in `get_passphrase()` → returns `Result<Zeroizing<String>, AgentError>`
   - L418 `self.cache.lock().unwrap().remove(...)` in `on_incorrect_passphrase()` → returns `()`
   - L487 `self.cached.lock().unwrap()` in `get_passphrase()` → returns `Result<Zeroizing<String>, AgentError>`

   For the `Result`-returning functions (L392, L409, L487): replace with `.lock().map_err(|_| AgentError::MutexError)?` — `AgentError::MutexError` already exists.
   For the `()`-returning functions (L386 `clear_cache()`, L418 `on_incorrect_passphrase()`): these are trait methods. Options: (a) change trait to return `Result` (breaking change to all implementors), or (b) silently handle poisoned mutex with `.lock().unwrap_or_else(|e| e.into_inner())` which recovers the inner data. **Prefer option (b)** — poisoned mutex recovery is appropriate here since the cached data is not safety-critical, and avoids a trait signature change.

   **Other sites**:
   - `crypto/secp256k1.rs:94` — `SigningKey::from_bytes(field_bytes).expect(...)`. Replace with `?` operator, propagate as crypto error.
   - `witness/server.rs:212,218,228` — `expect()` in receipt creation/signing. `create_receipt()` is a **private method** with only **1 production call site** (L493) and **6 test call sites** (L898, 908, 909, 912, 913, 921) — all in the same file. Change return type to `Result`, update all 7 call sites. Blast radius is contained to `server.rs`.

2. **Acceptable `expect()` sites** (document with `#[allow]` + comment):
   - `crypto/provider_bridge.rs:18` — `Lazy::new()` runtime creation. Tokio runtime failure is genuinely fatal. Add `#[allow(clippy::expect_used)]` with rationale comment.
   - `api/runtime.rs:590` — `expect("Length checked")` after explicit 32-byte validation. Add `#[allow(clippy::expect_used)]` with rationale.
   - `api/ffi.rs:200` — deprecated function. Add `#[allow(clippy::expect_used)]`.

3. **Add lint enforcement** to library crate roots:
   - `crates/auths-core/src/lib.rs`: add `#![deny(clippy::unwrap_used)]` and `#![deny(clippy::expect_used)]`
   - `crates/auths-sdk/src/lib.rs`: add `#![deny(clippy::unwrap_used)]` and `#![deny(clippy::expect_used)]`
   - Add `allow-unwrap-in-tests = true` to workspace `clippy.toml` if not already present

4. **Verify** no new `unwrap()`/`expect()` in auths-sdk (research found it clean, but verify after other tasks' changes)

### Scope decisions

- `crypto/encryption.rs:256,258,260` — `.try_into().unwrap()` after slicing. Safe-by-invariant. Replace with `.try_into().expect("slice length matches array size")` + `#[allow(clippy::expect_used)]`, OR replace with `?` if the function already returns `Result`.
- Deprecated FFI functions: add `#[allow]` rather than refactoring deprecated code.
- `clear_cache()` and `on_incorrect_passphrase()`: use `unwrap_or_else(|e| e.into_inner())` to avoid trait signature change.
