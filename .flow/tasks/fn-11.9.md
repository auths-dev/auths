## Add CI for FFI cross-compilation and binary size tracking

### Context

`auths-mobile-ffi` and `auths-verifier-swift` are excluded from the workspace. The CI at `.github/workflows/ci.yml` only runs workspace-level commands. There are no CI steps for:
- Cross-compiling to iOS or Android targets
- Testing the FFI crates
- Tracking binary size
- Running build scripts

Drift between FFI crates and workspace crates is undetectable until manual testing.

### What to Do

1. Add a CI job (or extend existing) for the unified `auths-ffi` crate:
   - `cargo check --target aarch64-apple-ios` (macOS runner)
   - `cargo check --target aarch64-apple-ios-sim` (macOS runner)
   - `cargo test` (host target, for inline tests)
   - `cargo clippy` and `cargo fmt --check`

2. Add Android cross-compilation check (if NDK is available on CI runner):
   - Install `cargo-ndk`
   - `cargo ndk -t arm64-v8a check`

3. Add binary size tracking:
   - Build release staticlib for iOS: `cargo build --release --target aarch64-apple-ios`
   - Record the size of `libauths_ffi.a`
   - Post size as a CI artifact or comment on PRs
   - Set a threshold alert (e.g., > 5MB stripped)

4. Add binding generation check:
   - Run `uniffi-bindgen generate --library ... --language swift --out-dir ...`
   - Verify generated Swift code compiles (if swift toolchain available)
   - At minimum, verify bindgen doesn't error

5. Ensure existing WASM check still runs:
   - `cd crates/auths-verifier && cargo check --target wasm32-unknown-unknown --no-default-features --features wasm`

### Key Files

- `.github/workflows/ci.yml` — existing CI workflow
- `crates/auths-ffi/Cargo.toml` — unified FFI crate (from fn-11.6)

### Constraints

- iOS cross-compilation requires macOS runner
- Android cross-compilation requires NDK (may need setup step)
- Binary size measurement must be reproducible (same release profile, same target)
- Don't block PRs on binary size — use it as an informational metric initially
