## Add pairing session SDK workflows (create, poll, respond)

### Context

The pairing flow has the most HTTP orchestration in the CLI:

- `online.rs` (lines 20-267): POST to create session, GET to poll status, WebSocket for real-time updates
- `join.rs` (lines 35-63, 228-241): GET session by code, POST response

The WebSocket transport stays in the CLI (presentation-layer optimization). The SDK will own the HTTP-based session lifecycle.

### What to Do

**1. Extend `PairingError` in `crates/auths-sdk/src/pairing.rs`** (or `error.rs`):

Add network-aware variants:
```rust
#[error("failed to create pairing session")]
SessionCreationFailed(#[source] NetworkError),

#[error("pairing session not found")]
SessionNotFound,

#[error("pairing session expired")]
SessionExpired,

#[error("failed to submit pairing response")]
ResponseSubmissionFailed(#[source] NetworkError),
```

**2. Create pairing session result types:**

```rust
pub struct PairingSession {
    pub id: String,
    pub code: String,
    pub endpoint: String,
    // ... fields from the session creation response
}

pub enum PairingSessionStatus {
    Waiting,
    Responded(PairingResponseData),
    Expired,
}
```

**3. Create SDK workflow functions in `crates/auths-sdk/src/workflows/pairing.rs`:**

- `create_online_session(registry_client: &impl RegistryClient, registry_url: &str, session_request: &SessionRequest) -> Result<PairingSession, PairingError>`
- `poll_session_status(registry_client: &impl RegistryClient, registry_url: &str, session_id: &str) -> Result<PairingSessionStatus, PairingError>`
- `fetch_session_by_code(registry_client: &impl RegistryClient, registry_url: &str, code: &str) -> Result<PairingSession, PairingError>`
- `submit_pairing_response(registry_client: &impl RegistryClient, registry_url: &str, session_id: &str, response: &PairingResponsePayload) -> Result<(), PairingError>`

Each follows the `registration.rs` pattern.

**4. Handle clock injection:**

- `join.rs` uses `Utc::now()` at lines 109, 114 for token expiry. Any time-dependent logic in the SDK must accept `now: DateTime<Utc>`.

### Key Files to Read First

- `crates/auths-cli/src/commands/device/pair/online.rs` -- session creation (lines 20-60), HTTP polling (lines 210-267)
- `crates/auths-cli/src/commands/device/pair/join.rs` -- session lookup (lines 35-63), response submission (lines 228-241)
- `crates/auths-sdk/src/pairing.rs` -- existing PairingError and pairing logic
- `crates/auths-sdk/src/registration.rs` -- pattern to follow

### Constraints

- No `anyhow`, no `reqwest`, no `Utc::now()` in SDK
- WebSocket logic stays in CLI -- only HTTP session CRUD moves to SDK
- All time-dependent logic accepts `now: DateTime<Utc>`
- `#[non_exhaustive]` on new enums
- Rustdoc on all public items

### Gotcha

`join.rs` line 35 creates `reqwest::Client::new()` with NO timeout -- this is a bug. When wiring up `HttpRegistryClient`, ensure it has a proper timeout configuration.
