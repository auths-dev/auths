## Description
## Consolidate reqwest usage and remove direct CLI dependency where possible

### Context

After fn-10.2, fn-10.4, and fn-10.6, all registry HTTP calls go through the SDK. The CLI should now have minimal direct `reqwest` usage.

### What to Do

**1. Audit remaining `reqwest` usage in `auths-cli`:**

Grep for `reqwest::` across the crate. Expected remaining uses:
- `services/oauth.rs` -- GitHub OAuth Device Flow (out of scope, stays in CLI)
- `services/gist.rs` -- GitHub Gist creation (out of scope, stays in CLI)
- `commands/device/pair/lan.rs` -- LAN health check (out of scope, stays in CLI)
- `factories/network.rs` -- `build_http_client()` to create the shared client
- `config.rs` -- `CliConfig.http_client` field

**2. If any non-exempt `reqwest` usage remains, extract it.**

**3. Evaluate whether `reqwest` can be removed from `auths-cli/Cargo.toml`:**

- If OAuth, Gist, and LAN still use it directly, keep the dep but document why
- If the `form` feature is only needed for OAuth, consider making it optional behind a feature flag
- At minimum, ensure there is no duplicate client construction -- all remaining uses should share the `CliConfig.http_client`

**4. Verify the composition root is clean:**

- `factories/mod.rs` should create `HttpRegistryClient` from the shared `reqwest::Client`
- All command handlers receive `HttpRegistryClient` (or `&impl RegistryClient`) through the call chain
- No ad-hoc `reqwest::Client` construction anywhere in the CLI

**5. Run full validation suite:**

```bash
cargo nextest run --workspace
cargo test --all --doc
cargo clippy --all-targets --all-features -- -D warnings
cargo fmt --check --all
cd crates/auths-verifier && cargo check --target wasm32-unknown-unknown --no-default-features --features wasm
```

### Constraints

- Do not remove reqwest if OAuth/Gist/LAN still need it
- Document any remaining direct reqwest usage with a code comment explaining why it stays
- No regressions on any platform (Linux, macOS, Windows)
## Acceptance
- [ ] No ad-hoc `reqwest::Client` construction in CLI command handlers (publish, claim, pairing)
- [ ] All registry HTTP calls go through `HttpRegistryClient` -> SDK workflow -> `RegistryClient` trait
- [ ] Remaining `reqwest` usage in CLI is limited to: OAuth, Gist, LAN health check, and the shared client factory
- [ ] `factories/mod.rs` creates `HttpRegistryClient` and threads it through the call chain
- [ ] `cargo nextest run --workspace` passes
