# fn-13.4 Abstract ssh-add subprocess behind port trait

## Description
Abstract the `Command::new("ssh-add")` subprocess call in `auths-core/src/api/runtime.rs` behind a port trait. This is the only subprocess call in auths-core production code.

**Note**: `register_keys_with_macos_agent_with_handle()` (the public function containing the `ssh-add` call) has **zero call sites** across the workspace — it is dead code from an external caller perspective. It is defined as a public API in `auths-core` but never called from the SDK, CLI, or any other crate. This task is still valuable for correctness (removing subprocess usage from a library crate), but the blast radius is minimal.

### What to do

1. **Create `SshAgentPort` trait** in `crates/auths-core/src/ports/ssh_agent.rs`:
   - `register_keys(&self, key_path: &Path) -> Result<(), SshAgentError>`
   - `SshAgentError` thiserror enum with `CommandFailed(String)`, `NotAvailable(String)`

2. **Register in `crates/auths-core/src/ports/mod.rs`**

3. **Refactor `crates/auths-core/src/api/runtime.rs`**:
   - `register_keys_with_macos_agent_with_handle()` and `register_keys_with_macos_agent_internal()` should accept an `&dyn SshAgentPort` parameter instead of directly spawning the subprocess
   - Since **no callers exist** outside this file, this is not a breaking change to any downstream crate. No call sites need updating.

4. **Create CLI adapter** in `crates/auths-cli/src/adapters/ssh_agent.rs`:
   - `PosixSshAgentAdapter` implementing `SshAgentPort`
   - Wraps `Command::new("ssh-add")` call
   - Gate with `#[cfg(target_os = "macos")]` (ssh-add registration is macOS-specific)
