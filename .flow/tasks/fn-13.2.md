# fn-13.2 Create SigningWorkflow with three-tier fallback

## Description
Create `SigningWorkflow` in `crates/auths-sdk/src/workflows/signing.rs` that encapsulates the three-tier commit signing fallback logic. This is the core extraction from `bin/sign.rs::run_sign()`.

### What to do

1. **Create `crates/auths-sdk/src/workflows/signing.rs`** with:
   - `CommitSigningParams` struct: `namespace: String`, `data: Vec<u8>`, `pubkey: Vec<u8>` (cached pubkey for agent signing), `repo_path: Option<PathBuf>`, `max_passphrase_attempts: usize` (default 3 — configurable for tests and future CLI flags)
   - `CommitSigningWorkflow` with `pub fn execute(ctx: &AuthsContext, params: CommitSigningParams, now: DateTime<Utc>) -> Result<String, SigningError>`
   - Three-tier internal flow:
     - **Tier 1** `try_agent_sign(ctx, params)` — calls `ctx.agent_signing.try_sign(namespace, pubkey, data)`. On `AgentSigningError::Unavailable` or `ConnectionFailed` → fall through. On `SigningFailed` → return fatal error. Returns SSHSIG PEM string (adapter already normalizes format).
     - **Tier 2** `try_auto_start_and_direct_sign(ctx, params, now)` — calls `ctx.agent_signing.ensure_running()` (best effort, ignore errors). Then runs passphrase retry loop (up to `params.max_passphrase_attempts`), decrypts key. On success, calls `ctx.agent_signing.add_identity()` as a **best-effort side effect** (errors logged, not fatal). Then falls through to Tier 3 direct sign with the already-decrypted seed. **Does NOT retry agent signing after adding identity** — this matches current `bin/sign.rs` behavior. Rationale: the key is already decrypted in memory, so direct signing is faster than another IPC round-trip. The identity is cached in the agent for *future* invocations (Tier 1 will succeed next time).
     - **Tier 3** `direct_sign(ctx, seed, params, now)` — calls `validate_freeze_state()` + `sign_with_seed()` from existing SDK signing module. Returns SSHSIG PEM string.

2. **Extend `SigningError`** in `crates/auths-sdk/src/signing.rs` with new variants:
   - `AgentUnavailable(String)`
   - `AgentSigningFailed(#[source] AgentSigningError)`
   - `PassphraseExhausted { attempts: usize }`
   - `KeychainUnavailable(String)`
   - `KeyDecryptionFailed(String)`
   - Add `#[non_exhaustive]` if not already present

3. **Passphrase retry logic**: The workflow owns the retry loop (up to `params.max_passphrase_attempts`). Calls `ctx.passphrase_provider.get_passphrase()`, attempts decryption, on failure calls `ctx.passphrase_provider.on_incorrect_passphrase()`. After exhaustion returns `SigningError::PassphraseExhausted { attempts }`.

4. **Register in `crates/auths-sdk/src/workflows/mod.rs`** — add `pub mod signing;`

5. **Write tests** in `crates/auths-sdk/tests/cases/signing.rs` (extend existing):
   - Test agent signing succeeds (Tier 1) — verify returns SSHSIG PEM
   - Test agent unavailable falls through to direct (Tier 3)
   - Test agent connection failed falls through to direct (Tier 3)
   - Test agent signing failed is fatal (does NOT fall through)
   - Test auto-start + direct sign (Tier 2 → 3)
   - Test passphrase exhaustion after N attempts (use `max_passphrase_attempts: 1` for speed)
   - Test freeze validation blocks signing
   - Test add_identity failure after auto-start is non-fatal (best effort)
   - Use `FakeAgentProvider` from fn-13.1

### Design decisions

- Accepts `now: DateTime<Utc>` — clock injection per CLAUDE.md policy
- Returns `String` (SSHSIG PEM) — CLI handles file I/O
- Uses `&AuthsContext` pattern (like `sign_artifact_attestation`), not generics
- Does NOT handle `AUTHS_PASSPHRASE` env var — that's the `PassphraseProvider` impl's concern
- Does NOT handle pubkey caching — CLI passes cached pubkey via `CommitSigningParams.pubkey`
- `max_passphrase_attempts` is configurable (default 3) for testing and future CLI configurability
- Both `try_sign` and `sign_with_seed` return the same SSHSIG PEM format — no normalization needed in workflow
