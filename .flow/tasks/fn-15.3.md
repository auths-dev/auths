## Description
## Fix auths-id production expect/unwrap calls

Fix ~6 production-code expect()/unwrap() calls in auths-id. These are the only non-test violations.

### Locations and fixes

1. **`src/witness.rs:44`** — `.expect("git2::Oid is always 20 bytes")`
   - **Fix**: `#[allow(clippy::expect_used)]` with `// INVARIANT: git2::Oid is always 20 bytes`
   - Rationale: Type-invariant guarantee from git2 library

2. **`src/witness.rs:61`** — `.expect("EventHash is always 20 bytes")`
   - **Fix**: `#[allow(clippy::expect_used)]` with `// INVARIANT: EventHash is always 20 bytes`
   - Rationale: Type-invariant guarantee from our own type

3. **`src/identity/resolve.rs:162`** — `.expect("ed25519_to_did_key requires exactly 32 bytes")`
   - **Fix**: Change function signature from `fn ed25519_to_did_key(public_key: &[u8]) -> String` to `fn ed25519_to_did_key(public_key: &[u8; 32]) -> String`
   - Rationale: Makes it impossible to pass wrong-length input; avoids semver-breaking Result change; the `try_into()` becomes infallible
   - **Impact**: Find all callers of `ed25519_to_did_key` and update call sites. Most already have `[u8; 32]` values.

4. **`src/identity/resolve.rs:77` and `:111`** — `.unwrap()` on `strip_prefix("did:keri:")`
   - **Fix**: Refactor to `let-else`: `let Some(suffix) = did.strip_prefix("did:keri:") else { ... }`
   - Rationale: Guarded by `starts_with()` check above but clippy can't prove it

5. **`src/keri/incremental.rs:283`** — `.expect("parent_count was 1")`
   - **Fix**: `#[allow(clippy::expect_used)]` with `// INVARIANT: parent_count==1 verified on line above`
   - Rationale: Logically infallible given preceding check

6. **`src/attestation/attestation_message.rs:27`** — `.unwrap()` on `previous` after `is_none()` check
   - **Fix**: Refactor to `if let Some(prev) = previous { ... }` pattern
