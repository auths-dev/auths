## Description
# Task: Spec-Compliant SAID Computation

Implement SAID computation using the KERI spec's 44-character `#` placeholder. This differs from the existing `compute_said` which uses empty strings.

## CRITICAL: Use serde_json, NOT json-canon

The existing codebase computes SAIDs with `serde_json::to_vec` (preserves struct field declaration order). `json-canon` (RFC 8785) sorts keys lexicographically, which would produce **incompatible** hashes. The KERI SAID spec requires insertion-order preservation.

References:
- `auths-verifier/src/keri.rs:615` -- `serde_json::to_vec` for signing payload
- `auths-core/src/crypto/said.rs:25-29` -- `compute_said` uses `serde_json`

## How this differs from existing `compute_said`

| Aspect | Current (`auths-crypto`) | Spec-compliant (`auths-keri`) |
|--------|--------------------------|-------------------------------|
| `d` field during hashing | Empty string `""` | 44x `#` placeholder |
| `x` field | Empty string `""` | Field removed entirely |
| `i` field (inception) | Empty string `""` | 44x `#` (self-certifying: `i == d`) |
| Serialization | `serde_json::to_vec` | `serde_json::to_vec` (same!) |

## What to do

1. Define `SAID_PLACEHOLDER` constant: 44 `#` characters

2. Implement `compute_spec_said(codec, event: &serde_json::Value)`:
   - Clone event object
   - Set `d` to 44x `#` placeholder
   - For `t == "icp"`: also set `i` to placeholder
   - Remove `x` field entirely
   - Serialize with `serde_json::to_vec` (NOT json-canon)
   - Blake3 hash the bytes
   - CESR-encode digest via `codec.encode_digest()`
   - Return 44-char SAID string

3. Implement `verify_spec_said(codec, event)`:
   - Extract `d` field value
   - Compute SAID via `compute_spec_said`
   - Compare

4. Write tests:
   - SAID is exactly 44 chars, starts with `E`
   - SAID is deterministic (same input -> same output)
   - SAID ignores `x` field (events with/without `x` produce same SAID)
   - Inception events get `i` placeholder applied
