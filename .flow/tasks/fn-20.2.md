# fn-20.2 Move protocol types and logic from auths-core to new crate

## Description
## Move protocol types and logic from auths-core to new crate

Move the transport-agnostic pairing protocol code from `auths-core/src/pairing/` to `auths-pairing-protocol`.

### Files to move

From `crates/auths-core/src/pairing/`:
- `token.rs` → `crates/auths-pairing-protocol/src/token.rs`
- `response.rs` → `crates/auths-pairing-protocol/src/response.rs`
- `types.rs` → `crates/auths-pairing-protocol/src/types.rs`

### Crypto bridge resolution

`token.rs` uses (1 call):
- `crate::crypto::provider_bridge::verify_ed25519_sync(pubkey, msg, sig)` in `PairingSession::verify_response()`

`response.rs` uses (2 calls):
- `provider_bridge::sign_ed25519_sync(seed, msg)` in `PairingResponse::create()`
- `provider_bridge::verify_ed25519_sync(pk, msg, sig)` in `PairingResponse::verify()`

These are thin sync wrappers over `RingCryptoProvider`. Replace with direct `auths_crypto` calls. Options:
1. Add sync Ed25519 functions to `auths-crypto` (cleanest — avoids tokio dep)
2. Use `ring` directly like `auths-mobile-ffi` does
3. Add a sync helper in the new crate that creates a `RingCryptoProvider` and blocks on the async call

Prefer option 1 or 2 — the protocol crate should NOT depend on tokio.

### types.rs changes

`types.ts` has one field typed as `auths_verifier::types::DeviceDID`. Replace with `String`.

**Why String, not DeviceDID:** Adding auths-verifier as a dependency is technically safe (same Batch 2, no circular dep — auths-verifier doesn't depend back on this crate). However, it defeats the purpose of extraction: mobile apps pulling in the protocol crate would transitively get the entire auths-verifier library. The protocol crate should have the minimal dependency chain: only `auths-crypto` + external crates. The DID is just a string identifier at the protocol level.

Also: `types.rs` currently derives `schemars::JsonSchema` on all types. Remove this derive in the new crate — these are protocol types, not API documentation types. The CLI/server can add OpenAPI annotations separately.

### Module structure after move

```
crates/auths-pairing-protocol/src/
  lib.rs        (pub mod + re-exports)
  error.rs      (from fn-20.1)
  token.rs      (PairingToken, PairingSession, normalize_short_code)
  response.rs   (PairingResponse)
  types.rs      (Base64UrlEncoded, SessionStatus, request/response types)
```

### Test migration

Unit tests in token.rs and response.rs move with the code. Update crypto calls in tests to match the new approach.

### Files
- Move + modify: token.rs, response.rs, types.rs
- Modify: `crates/auths-pairing-protocol/src/lib.rs`

### Verification
```bash
cargo check -p auths-pairing-protocol
cargo nextest run -p auths-pairing-protocol
```

## Move protocol types and logic from auths-core to new crate

Move the transport-agnostic pairing protocol code from `auths-core/src/pairing/` to `auths-pairing-protocol`.

### Files to move

From `crates/auths-core/src/pairing/`:
- `token.rs` → `crates/auths-pairing-protocol/src/token.rs`
- `response.rs` → `crates/auths-pairing-protocol/src/response.rs`
- `types.rs` → `crates/auths-pairing-protocol/src/types.rs`

**Keep in auths-core:**
- `qr.rs` (presentation concern — terminal QR rendering)
- `mod.rs` (becomes re-export facade in fn-20.3)

### Crypto bridge resolution

`token.rs` and `response.rs` use `crate::crypto::provider_bridge` for Ed25519 sign/verify. Replace with direct `auths_crypto` calls:

- `provider_bridge::sign(seed, message)` → use `auths_crypto` Ed25519 signing
- `provider_bridge::verify(public_key, message, signature)` → use `auths_crypto` Ed25519 verify

Check the exact `auths_crypto` public API and map the calls. The goal is functional parity — same inputs, same outputs, just different import paths.

### Module structure after move

```
crates/auths-pairing-protocol/src/
  lib.rs        (pub mod + re-exports)
  error.rs      (from fn-20.1)
  token.rs      (PairingToken, PairingSession)
  response.rs   (PairingResponse)
  types.rs      (PairingUri, ShortCode, etc.)
```

### Test migration

Move unit tests that are embedded in token.rs and response.rs along with the code. Tests that use `crate::crypto::provider_bridge` test helpers need to be updated to use `auths_crypto` test utilities.

### Files
- Move + modify: token.rs, response.rs, types.rs
- Modify: `crates/auths-pairing-protocol/src/lib.rs` (add modules)
