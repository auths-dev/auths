## Wire policy evaluation into OidcIssuer::exchange()

Insert policy evaluation into the token exchange flow. After chain verification and before JWT issuance, optionally evaluate a policy against the OIDC claims.

### Integration point

`crates/auths-oidc-bridge/src/issuer.rs` — `OidcIssuer::exchange()` method.

Current flow:
1. Decode root public key
2. Validate audience
3. Validate TTL
4. Verify attestation chain
5. Check validity window
6. Extract capabilities ← INSERT POLICY CHECK HERE
7. Build JWT claims
8. Sign JWT

### Implementation

The `OidcIssuer` needs access to the compiled policy. Two options:
- Pass it as a parameter to `exchange()` (cleaner, testable)
- Store a reference in `OidcIssuer` (requires lifetime or Arc)

Prefer passing from `BridgeState` at the handler call site, keeping `OidcIssuer` focused on JWT mechanics:

```rust
// In the exchange handler (routes.rs or equivalent):
let policy = state.workload_policy(); // Option<&CompiledPolicy>
let response = issuer.exchange(&request, policy, ...)?;

// In OidcIssuer::exchange(), after step 5:
#[cfg(feature = "oidc-policy")]
if let Some(policy) = workload_policy {
    let now_dt = DateTime::from_timestamp(now_epoch, 0)
        .expect("u64 epoch seconds always fits in DateTime");
    let ctx = build_eval_context_from_oidc(&claims, now_dt)?;
    let decision = evaluate_strict(policy, &ctx);
    if decision.outcome != Outcome::Allow {
        return Err(BridgeError::PolicyDenied {
            reason: decision.message.unwrap_or_else(|| "policy denied".into()),
        });
    }
}
```

### Clock conversion

The bridge's `ClockFn` returns `u64` epoch seconds. Convert to `DateTime<Utc>` via `DateTime::from_timestamp(epoch, 0)`. This only returns `None` for values exceeding `i64::MAX` seconds (~292 billion years) — use `.expect()` with an INVARIANT comment rather than adding a `BridgeError::InvalidClock` variant for dead code.

### Ordering concern

Policy evaluation needs the `OidcClaims` struct, but the current flow builds claims at step 7 (after capability extraction). The adapter function needs: `keri_prefix`, `sub`, `capabilities`, `github_*` fields. These are all available after step 5 from the chain and request — build a pre-claims struct or call the adapter with the raw fields before constructing `OidcClaims`. Alternatively, build `OidcClaims` earlier and evaluate against it.

### Logging

Log at `info` level when a policy evaluation denies a request (include the decision reason).
