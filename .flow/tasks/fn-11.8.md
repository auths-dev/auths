## Fix clock injection violations in FFI code

### Context

The project enforces clock injection: `Utc::now()` is banned in core/SDK code. All time-sensitive functions accept `now: DateTime<Utc>` as a parameter. However, the FFI crate violates this:
- `create_pairing_response()` at line 677 of `crates/auths-mobile-ffi/src/lib.rs` uses `std::time::SystemTime::now()` directly.

### What to Do

1. Audit all FFI code for direct clock access (`Utc::now()`, `SystemTime::now()`, `Instant::now()`).

2. Replace with clock injection. Two options:
   - **Option A** (preferred): Accept `now_unix_secs: u64` as a parameter in FFI functions that need time. The mobile app provides the current time. Simple, no trait objects.
   - **Option B**: Use `ClockProvider` via callback interface. More complex but consistent with core pattern.

3. For the unified `auths-ffi` crate: decide on a convention and apply consistently.

4. Ensure all expiry calculations, timestamp comparisons, and time-based logic use the injected time.

### Key Files

- `crates/auths-mobile-ffi/src/lib.rs:677` — SystemTime::now() violation
- `crates/auths-verifier/src/clock.rs:17-25` — ClockProvider trait
- `crates/auths-core/src/ports/clock.rs` — ClockProvider re-export

### Constraints

- Follow CLAUDE.md: "Never add `Utc::now()` to domain or core logic — inject it instead"
- The CLI calls `Utc::now()` at the presentation boundary — FFI is also a presentation boundary, so Option A (parameter injection) is consistent
- Must be testable: deterministic time in tests
## Acceptance
- [ ] No `SystemTime::now()` or `Utc::now()` calls in FFI crate code
- [ ] Time-dependent FFI functions accept `now_unix_secs: u64` parameter
- [ ] Tests verify expiry logic with deterministic injected time
- [ ] Consistent with CLAUDE.md clock injection pattern
