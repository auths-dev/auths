## Refactor CLI pairing handlers to use SDK pairing workflows

### Context

After fn-10.5, the SDK has pairing session functions. Now thin out the CLI handlers.

### What to Do

**1. Update `crates/auths-cli/src/commands/device/pair/online.rs`:**

- Replace inline session creation HTTP call with `sdk::create_online_session()`
- Replace HTTP polling loop (lines 210-267) with `sdk::poll_session_status()`
- Keep WebSocket fast-path in CLI (lines 158-209) -- it calls `poll_session_status` as fallback
- Keep all spinner updates, QR code display, and `console::style` formatting in CLI

**2. Update `crates/auths-cli/src/commands/device/pair/join.rs`:**

- Remove `reqwest::Client::new()` at line 35
- Replace GET session-by-code with `sdk::fetch_session_by_code()`
- Replace POST response with `sdk::submit_pairing_response()`
- Keep `rpassword::prompt_password` and all terminal I/O in CLI
- Wire up `HttpRegistryClient` from CLI composition root

**3. Wire up `HttpRegistryClient` in pairing handlers:**

- The `handle_pair()` function at `pair/mod.rs:69` currently takes `http_client: &reqwest::Client`. Change it to take `registry_client: &impl RegistryClient` (or `&HttpRegistryClient` concretely if simpler).
- Update `handle_initiate_online()` and `handle_join_online()` signatures similarly.
- Update `device/authorization.rs` which passes `http_client` through.

- Fix the no-timeout bug: `join.rs` used `reqwest::Client::new()` with no timeout. The `HttpRegistryClient` from `auths-infra-http` should have proper timeout config.
