# fn-19: Batch Git Storage

**Priority:** P2 — Scale enabler
**Effort:** 2-3 weeks

## Overview

The current `GitRegistryBackend` creates one commit per KEL event. For registries with thousands of identities, batch ingestion (e.g., federation sync) is O(n) commits. This epic adds a `batch_append_events` method that stages all mutations in a single Git commit.

## Scope

1. Implement `batch_append_events` as an **inherent method** on `GitRegistryBackend` (NOT on the frozen `RegistryBackend` trait)
2. In-memory state overlay for validating sequential events within a batch
3. Concurrent batch writer tests
4. Batch benchmarks (1000 events: batch vs sequential)

## Approach

### Trait freeze

The `RegistryBackend` trait at `crates/auths-id/src/storage/registry/backend.rs:285-292` is explicitly frozen. `batch_append_events` is an inherent method on `GitRegistryBackend` only. No trait changes.

### Exact types

- **Prefix:** `auths_verifier::keri::Prefix` — newtype around `String`, validated to start with `'E'` (Blake3-256 derivation code)
- **Event:** `auths_id::keri::event::Event` — enum with `Icp(IcpEvent)`, `Rot(RotEvent)`, `Ixn(IxnEvent)` variants
- **State:** `auths_id::keri::state::KeyState` — contains `current_keys`, `next_commitment`, `sequence`, `last_event_said`, `is_abandoned`, `threshold`, `next_threshold`
- **Cached state:** `CachedStateJson { version, state: KeyState, validated_against_said }`

### Per-event writes (3 files per event)

Each event produces 3 blob writes via `TreeMutator::write_blob()`:
1. `event.json` — the raw event
2. `tip.json` — latest event SAID and sequence
3. `state.json` — computed `CachedStateJson` with full `KeyState`

For a batch of 1000 events: up to 3000 `write_blob()` calls, then a single `build_tree()` and `create_commit()`.

### State computation within batch

State transitions use existing `KeyState` methods:
- `KeyState::apply_rotation()` — updates keys, next commitment, thresholds, sequence, SAID
- `KeyState::apply_interaction()` — updates sequence and SAID only

The in-memory overlay is `HashMap<Prefix, KeyState>`. For the first event of a prefix in the batch, read current state from Git. For subsequent events on the same prefix, use the overlay.

### Lock and CAS

Current `create_commit` handles CAS internally. For batch:
1. Acquire `AdvisoryLock` (POSIX flock on `registry.lock`)
2. Read tip commit ref
3. Validate + stage all events
4. `TreeMutator::build_tree()` with base tree from tip
5. Create commit (with CAS: verify tip ref hasn't changed since step 2)
6. Lock released on drop

Lock held for entire batch — blocks other writers. Acceptable tradeoff for atomicity.

### Batch semantics

- **Atomic:** if event N fails validation, entire batch is rejected. No partial commits.
- **Mixed-prefix:** a batch can contain events for different identities.
- **Same-prefix sequential:** a batch can contain ICP at seq 0 then IXN at seq 1 for the same prefix. The overlay handles this.
- **Empty batch:** no-op, returns `Ok(())`.
- **Error:** `RegistryError::BatchValidationFailed { index: usize, source: Box<RegistryError> }` includes the failing event index.

## Risks / Dependencies

- Long lock hold for large batches blocks concurrent writers
- In-memory overlay must exactly match what sequential `append_event` would produce
- SQLite index (`indexed-storage` feature): update after Git commit, not inline
- Metadata counters (`identity_count`, `device_count`) tracked in-memory during batch
