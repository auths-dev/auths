# FFI & Mobile Readiness

## Overview

Make the auths core bindable via uniffi for iOS and Android mobile apps. The current architecture uses `Arc<dyn Trait>` trait objects extensively (`AuthsContext` has 9 such fields), has async coupling through `CryptoProvider` and tokio hard-dependencies, and the existing `auths-mobile-ffi` crate duplicates crypto logic instead of reusing `auths-crypto`. This epic creates clean, synchronous, standard-data-type FFI boundaries that mobile clients can bind without implementing Rust traits directly or pulling in tokio/git2.

## Scope

### In Scope
- Sync CryptoProvider variant for native targets (auths-crypto)
- Feature-gating tokio in auths-core so mobile FFI compiles without it
- Eliminating crypto duplication in auths-mobile-ffi by depending on auths-crypto
- FFI-safe facade layer with concrete types replacing Arc<dyn Trait> at the boundary
- RegistryBackend FFI adapter (Vec<T> returns instead of FnMut visitors)
- uniffi callback_interface for platform-provided traits (keychain, passphrase)
- Unifying FFI crates (auths-mobile-ffi + auths-verifier-swift) and upgrading to uniffi 0.31
- Zeroization hardening at FFI boundaries (opaque handles for key material)
- Clock injection fixes (SystemTime::now() violations)
- CI for FFI cross-compilation (iOS, Android build checks)
- Binary size tracking baseline

### Out of Scope
- Full Android Keystore JNI implementation (use uniffi callback_interface instead â€” Kotlin implements)
- HTTP-backed RegistryBackend implementation for mobile (separate epic)
- SQLite-backed IdentityStorage for mobile (separate epic)
- Mobile app UI/UX
- CocoaPod or SPM distribution packaging
- Kotlin Multiplatform support

## Approach

### Architecture Decision: Layered FFI Facade

The mobile FFI will be a **stateful SDK** that wraps `AuthsContext` behind concrete facades, not a stateless function library. The key insight: mobile apps should provide platform services (keychain, passphrase UI) via uniffi callback interfaces, while Rust owns the crypto, verification, and identity logic.

```
Mobile App (Swift/Kotlin)
    |
    | uniffi callback_interface (keychain, passphrase prompt)
    v
auths-ffi (uniffi facade)
    |
    | concrete types, synchronous calls
    v
auths-sdk (AuthsContext with injected impls)
    |
    v
auths-core (sync crypto)  +  auths-id (identity/storage)
    |
    v
auths-crypto (SyncCryptoProvider + RingCryptoProvider)
```

### Key Design Decisions

1. **SyncCryptoProvider trait**: Add a synchronous counterpart to the async `CryptoProvider` in `auths-crypto`. `RingCryptoProvider` implements both. The async variant remains for WASM (`WebCryptoProvider`). The FFI path uses the sync trait directly â€” no tokio runtime needed.

2. **Feature-gated tokio**: `auths-core` gets a `async-runtime` feature (default on) gating tokio and `provider_bridge.rs`. Without it, auths-core compiles as a pure sync library. Mobile FFI depends on `auths-core` with `default-features = false`.

3. **FFI facade pattern**: A new module in the unified FFI crate provides concrete structs (`MobileContext`, `MobileVerifier`, `MobileIdentityManager`) that internally construct `AuthsContext` from configuration + callback interfaces. Mobile apps never see `Arc<dyn Trait>`.

4. **RegistryBackend adapter**: A `SimpleRegistryAdapter` wraps `RegistryBackend` and replaces `visit_*(&mut dyn FnMut)` methods with `list_*() -> Vec<T>` methods. This is the FFI-visible surface. The full `RegistryBackend` trait remains unchanged internally.

5. **Callback interfaces for platform services**: `PassphraseProvider` and `KeyStorage` get uniffi `callback_interface` variants so Swift uses iOS Keychain Services and Kotlin uses Android Keystore natively.

6. **Key material never crosses FFI as strings**: Replace `current_key_pkcs8_hex: String` with opaque handles (`KeyHandle(u64)`) that reference keys stored in Rust-side `Zeroizing<Vec<u8>>` storage. Mobile apps pass handles to subsequent operations.

7. **Unified FFI crate**: Merge `auths-mobile-ffi` and `auths-verifier-swift` into a single `auths-ffi` crate on uniffi 0.31 with proc-macro approach.

### Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| ring cross-compilation failures on Android NDK | Medium | High | Test early in Task 4; ring 0.17.14 has ARM64 assembly |
| uniffi 0.31 breaking changes from 0.28 | Medium | Medium | Migrate incrementally; test generated Swift/Kotlin compiles |
| Feature-gating tokio breaks auths-core consumers | Low | High | Default feature `async-runtime` on; only mobile FFI opts out |
| SyncCryptoProvider adds maintenance burden | Low | Low | RingCryptoProvider already wraps sync ring calls; impl is trivial |
| Binary size exceeds App Store limits | Low | Medium | Track size in CI; strip symbols, LTO, opt-level "z" |
| Opaque key handles leak if app crashes | Medium | Medium | Implement handle table with weak refs and TTL |

### Alternatives Considered

1. **Keep stateless function library**: Simpler but forces mobile apps to reimplement storage, state management, and workflow orchestration. Rejected because it defeats the purpose of an SDK.

2. **Use cxx instead of uniffi**: Better performance but only generates C++ bindings, not Swift/Kotlin. Rejected for mobile use case.

3. **Make CryptoProvider conditionally sync via cfg**: Would require `#[cfg(not(target_arch = "wasm32"))]` everywhere. Rejected in favor of a separate sync trait which is cleaner.

4. **Keep separate FFI crates**: Causes symbol duplication when linking both, version drift risk, and duplicated build infrastructure. Rejected.

## Acceptance Criteria

- [ ] `auths-crypto` exports a `SyncCryptoProvider` trait; `RingCryptoProvider` implements it
- [ ] `auths-core` compiles with `--no-default-features` (no tokio)
- [ ] `auths-mobile-ffi` crypto duplication eliminated â€” depends on `auths-crypto`
- [ ] Unified `auths-ffi` crate with uniffi 0.31, proc-macro approach
- [ ] FFI facade types exist: `MobileContext`, `MobileVerifier` (concrete, no trait objects)
- [ ] `PassphraseProvider` and `KeyStorage` exposed as uniffi callback_interfaces
- [ ] `RegistryBackend` adapter uses `Vec<T>` returns (no FnMut visitors at FFI boundary)
- [ ] Key material uses opaque handles, not hex strings
- [ ] No `SystemTime::now()` calls in FFI crate (clock injection enforced)
- [ ] CI checks: iOS (`aarch64-apple-ios`) and Android (`aarch64-linux-android`) cross-compilation
- [ ] Binary size baseline recorded in CI
- [ ] All existing workspace tests pass (`cargo nextest run --workspace`)
- [ ] WASM target still compiles (`auths-verifier` with `--features wasm`)

## Non-Functional Targets

- **Binary size**: FFI staticlib < 5MB stripped (iOS arm64)
- **Build time**: Cross-compilation < 5 minutes on CI
- **API surface**: < 20 exported uniffi functions for initial mobile SDK
- **Thread safety**: All exported types are Send + Sync (uniffi requirement)
