# fn-20: LAN Pairing Protocol Extraction

**Priority:** P3 — Mobile enabler
**Effort:** 1 week

## Overview

Extract the transport-agnostic pairing protocol from `auths-core/src/pairing/` into a standalone `auths-pairing-protocol` crate. Mobile apps (iOS/Android) can use pairing without pulling in auths-core's heavy dependencies (axum, tower-http, mdns-sd, qrcode, platform keychains).

**Backwards compatibility is NOT a concern** — we are pre-launch. All consumers (CLI, SDK) update their imports directly.

## Scope

1. Create `crates/auths-pairing-protocol/` with protocol types, errors, and logic
2. Move protocol code from auths-core to new crate
3. Delete pairing code from auths-core (no re-export facade — pre-launch, just update imports)
4. Update CLI and SDK imports
5. Add protocol state machine
6. Update `scripts/releases/2_crates.py` publish ordering

## Approach

### Dependency tree placement

`auths-pairing-protocol` depends on `auths-crypto` (Batch 1). Goes in **Batch 2**:

```python
PUBLISH_BATCHES = [
    ["auths", "auths-crypto", "auths-index", "auths-policy", "auths-telemetry"],  # Batch 1
    ["auths-verifier", "auths-keri", "auths-pairing-protocol"],  # Batch 2
    ...
]
```

### Dependencies (minimal)

- `auths-crypto` — `RingCryptoProvider`, `SecureSeed`, `CryptoError`
- `x25519-dalek` — X25519 key exchange
- `rand` — ephemeral key generation
- `base64` — token encoding
- `serde`, `serde_json` — serialization
- `zeroize` — key material cleanup
- `chrono` — expiry timestamps
- `thiserror` — error types

**NOT included:** `qrcode`, `axum`, `tower-http`, `mdns-sd`, `reqwest`, `schemars`.

### What moves vs. stays

**Moves to `auths-pairing-protocol`:**
- `token.rs` — `PairingToken`, `PairingSession`, `normalize_short_code()`
- `response.rs` — `PairingResponse`
- `types.rs` — `Base64UrlEncoded`, `SessionStatus`, request/response types (but replace `DeviceDID` with `String` to avoid depending on auths-verifier)
- Protocol error variants from `error.rs`

**Stays in auths-core:**
- `qr.rs` — presentation concern (terminal QR rendering, `qrcode` crate)

**Deleted from auths-core (no re-export facade):**
- `mod.rs`, `error.rs`, `token.rs`, `response.rs`, `types.rs` — removed after extraction
- auths-core keeps only `qr.rs` under a `pairing` module (or moves QR to a utility module)

### Error type split (16 total variants)

**Protocol errors (new crate — `ProtocolError`):**
`Expired`, `InvalidSignature`, `SessionConsumed`, `KeyExchangeFailed`, `KeyGenFailed`, `RngFailed`, `InvalidUri`, `InvalidShortCode`, `ShortCodeNotFound`, `Serialization`

**Transport errors (stay as `PairingError` in CLI/wherever needed):**
`RelayError`, `LocalServerError`, `MdnsError`, `NoPeerFound`, `LanTimeout`, `QrCodeFailed`

Transport errors are defined where they're used (CLI transport layer), not in a shared crate.

### Crypto bridge resolution

`token.rs` and `response.rs` use `crate::crypto::provider_bridge::{sign_ed25519_sync, verify_ed25519_sync}`. These are thin sync wrappers over `RingCryptoProvider`.

The new crate uses `RingCryptoProvider` directly (sync calls). `auths-mobile-ffi` already proves this pattern works — it uses `ring::signature::Ed25519KeyPair` directly without the async wrapper.

Specifically:
- `provider_bridge::sign_ed25519_sync(seed, msg)` → `RingCryptoProvider.sign_ed25519(seed, msg)` (needs tokio-free sync path or direct ring call)
- `provider_bridge::verify_ed25519_sync(pk, msg, sig)` → `RingCryptoProvider.verify_ed25519(pk, msg, sig)`

If `CryptoProvider` trait is async-only, add sync helper functions in `auths-crypto` or use `ring` directly (like auths-mobile-ffi does).

### types.rs DeviceDID dependency

`types.rs` uses `auths_verifier::types::DeviceDID` in one field. Since we don't want auths-pairing-protocol to depend on auths-verifier (would create layering issues — both are Batch 2), replace with `String`. The protocol crate is transport-agnostic and DID-agnostic.

### Consumer updates (pre-launch, no backward compat needed)

**auths-cli** (7 files in `commands/device/pair/`):
- Change `use auths_core::pairing::*` → `use auths_pairing_protocol::*`
- QR imports stay from auths-core: `use auths_core::pairing::qr::*`

**auths-sdk** (1 file: `src/pairing.rs`):
- Change `use auths_core::pairing::normalize_short_code` → `use auths_pairing_protocol::normalize_short_code`

## Risks / Dependencies

- `CryptoProvider` trait is async — need sync Ed25519 path in new crate (either add sync methods to auths-crypto or use ring directly)
- `DeviceDID` → `String` in types.rs changes the type signature
- auths-mobile-ffi reimplements pairing logic independently — should eventually be updated to use new crate (out of scope)
