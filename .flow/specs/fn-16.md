# OIDC-to-Policy Bridge

## Overview

Connect the OIDC bridge's token exchange flow to the policy engine so authorization policies can gate JWT issuance. The pieces exist independently — the bridge issues JWTs with capabilities, the policy engine evaluates capabilities — but there is no glue layer.

## Scope

- Add `auths-policy` as an optional dependency of `auths-oidc-bridge` behind an `oidc-policy` feature flag
- Create an adapter function in `auths-oidc-bridge` that converts `OidcClaims` into `EvalContext`
- Wire policy evaluation into `OidcIssuer::exchange()` between chain verification and JWT issuance
- Add `BridgeConfig` fields for policy source (path or inline JSON) — NOT `CompiledPolicy`
- Compile and store `CompiledPolicy` in `BridgeState` at startup
- Add `BridgeError` variants for policy denial

## Approach

### Architecture Decision: Adapter lives in `auths-oidc-bridge`

The adapter function (`OidcClaims → EvalContext`) MUST live in `auths-oidc-bridge`, not `auths-policy`. Reason: `OidcClaims` is defined in the bridge crate (`src/token.rs`). Putting the adapter in `auths-policy` would create a reverse dependency (policy → bridge), violating the dependency order (policy is Batch 1, bridge is Batch 3).

### Existing reuse points

- `EvalContext` builder API (`crates/auths-policy/src/context.rs`) — has `.workload_issuer()`, `.workload_claim()`, `.signer_type()`, `.repo()`, `.environment()`, `.capability()`
- `context_from_attestation()` (`crates/auths-id/src/policy/mod.rs`) — pattern to follow
- `evaluate_strict()` (`crates/auths-policy/src/eval.rs`) — core evaluator, use directly
- `CompiledPolicy` + `PolicyBuilder` (`crates/auths-policy/src/builder.rs`) — policy construction
- `BridgeError` (`crates/auths-oidc-bridge/src/error.rs`) — add new variants here
- Feature gating pattern: `github-oidc = ["dep:reqwest"]` in bridge Cargo.toml

### Config vs State separation

`BridgeConfig` holds the policy source (file path or inline JSON string). `BridgeState::new()` compiles the policy at startup and holds the `Option<CompiledPolicy>`. This keeps config serializable/cloneable and separates parse-time concerns from runtime state.

### Clock reconciliation

Policy uses `now: DateTime<Utc>`. Bridge uses `ClockFn = Arc<dyn Fn() -> u64>` (epoch seconds). The adapter converts via `DateTime::from_timestamp(epoch, 0)` — this only returns `None` for values exceeding `i64::MAX` seconds (~292 billion years), so use `.expect()` with an INVARIANT comment rather than a new error variant.

### EvalContext field mapping

| OidcClaims field | EvalContext field | Notes |
|------------------|-------------------|-------|
| `sub` (KERI DID) | `subject` | Parsed via `CanonicalDid::parse()` |
| `keri_prefix` | `issuer` | The identity that issued the attestation chain |
| `capabilities` | `capabilities` | Parsed via `CanonicalCapability::parse()` |
| `github_repository` | `repo` | Via `.repo()` builder |
| `github_actor` | `workload_claims["github_actor"]` | Via `.workload_claim()` |
| `target_provider` | `attrs["provider"]` | Via `.attr()` |
| (always) | `signer_type = Workload` | Via `.signer_type()` |

### Fail-safe default

When no policy is configured (`config.workload_policy` is `None`): **allow** — preserves backwards compatibility. Log a warning at bridge startup if no policy is configured.
