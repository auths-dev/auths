# Epic: KERI Compliance Translation Layer (auths-keri)

## Overview

Achieve wire-level interoperability with Trust over IP KERI (v0.9) by building a translation layer that converts between Auths' internal JSON event representation and spec-compliant CESR streams. The core codebase (`auths-id`, `auths-verifier`, `auths-core`) remains untouched. Git-native JSON storage remains the primary format. The translation layer sits alongside the existing stack and is opt-in.

## Scope

**In scope:**
- New `auths-keri` workspace crate at Layer 1-2
- CESR codec trait + `cesride`-backed implementation
- Spec-compliant SAID with 44-char `#` placeholder
- Dynamic KERI version strings (`KERI10JSON{hex_size}_`)
- Detached-signature event serialization
- CESR stream assembly (events + attachments)
- Bidirectional export/import (auths KEL <-> CESR stream)
- Text-domain CESR only (no binary domain in v1)
- JSON serialization only (no CBOR/MsgPack in v1)

**Out of scope:**
- `dip`/`drt` delegation events
- Multi-sig (`kt > 1`) with weighted thresholds
- OOBI generation/parsing
- TEL/ACDC
- KAWA witness agreement protocol details
- Binary domain CESR
- CBOR/MessagePack serialization

## Architecture

```
auths-cli / auths-sdk (unchanged)
  |
  +-- auths-id (unchanged -- internal JSON events, Git storage)
  |     |
  |     +-- KelPort trait (existing -- returns Vec<Event>)
  |
  +-- auths-keri (NEW -- this epic)
        |
        +-- codec.rs     -- CesrCodec trait + CesrV1Codec backed by cesride
        +-- said.rs      -- Spec-compliant SAID with # placeholder
        +-- version.rs   -- Dynamic KERI10JSON{size}_ version strings
        +-- event.rs     -- Detached-signature event serialization
        +-- stream.rs    -- Full CESR stream assembly (events + attachments)
        +-- roundtrip.rs -- Export auths KEL -> CESR, import CESR -> auths types
```

Layer placement: `auths-keri` depends on `auths-verifier` (Layer 1) and `auths-crypto` (Layer 0). It sits at Layer 1-2 alongside verifier. It does NOT depend on `auths-id` or `auths-core`.

## Critical Design Decisions (from gap analysis)

### 1. SAID serialization: serde_json, NOT json-canon

The existing codebase computes SAIDs using `serde_json::to_vec` which preserves struct field declaration order. `json-canon` (RFC 8785 JCS) sorts keys lexicographically, which would produce WRONG hashes incompatible with existing SAIDs.

- `auths-verifier/src/keri.rs:615` -- uses `serde_json::to_vec` for signing payload
- `auths-id/src/keri/validate.rs:368` -- uses `serde_json::to_vec` for SAID verification
- `auths-core/src/crypto/said.rs:25-29` -- `compute_said` uses `serde_json::to_vec`

**Decision**: Use `serde_json::to_vec` (not `json_canon::to_vec`) for SAID computation. Remove `json-canon` from dependencies.

### 2. Signature encoding: base64url, NOT hex

The `x` field in KERI events is base64url-no-pad encoded, NOT hex:
- `auths-verifier/src/keri.rs:661-663` -- `URL_SAFE_NO_PAD.decode(sig_str)`
- `auths-id/src/keri/validate.rs:424` -- `URL_SAFE_NO_PAD.decode(sig_str)`

**Decision**: Use `base64::engine::general_purpose::URL_SAFE_NO_PAD` for `x` field decode/encode.

### 3. Event type: `auths_verifier::keri::KeriEvent`

Two parallel event types exist:
- `auths_verifier::keri::KeriEvent` (Layer 1, `s: String`, decimal sequence)
- `auths_id::keri::event::Event` (Layer 3, `s: KeriSequence`, hex sequence)

**Decision**: Use `auths_verifier::keri::KeriEvent` as the input/output type. This keeps `auths-keri` at Layer 1-2 with minimal deps and avoids pulling in `auths-id`. The verifier types are the canonical definitions (re-exported by `auths-id`).

### 4. Dependencies: per-crate, not workspace

Only `base64`, `thiserror`, and internal crates (`auths-verifier`, `auths-crypto`) use `.workspace = true`. All others (`cesride`, `serde`, `serde_json`, `blake3`, `hex`) are per-crate explicit versions.

## Acceptance Criteria

- `cargo build -p auths-keri` compiles with zero warnings
- `cargo clippy -p auths-keri --all-targets -- -D warnings` passes
- CESR codec round-trips: `decode(encode(bytes)) == bytes` for Ed25519 keys, signatures, and Blake3 digests
- SAID output is 44 chars starting with `E`, deterministic, independent of `x` field
- Version string hex size matches actual serialized byte count
- CESR stream: JSON body + `-AAB` counter + 88-char indexed signature
- Round-trip: `import(export(events)) == events` for a 3-event KEL
