# Epic: Panic Eradication and Workspace Lint Enforcement

**Priority**: P0 — Ship blocker
**Revised Effort**: 3-5 days (down from 2-3 weeks — see scope reduction below)

## Overview

Enforce `deny(clippy::unwrap_used, clippy::expect_used)` workspace-wide to prevent panics in production code. A single `unwrap()` panic in a Git hook kills a developer's commit flow; in CI, it's a pipeline outage.

### Scope Reduction (Research Findings)

The original estimate assumed ~2,500 unwrap/expect calls needed fixing. **Actual production-code count is ~13:**

| Crate | Originally Estimated | Actual Production | Status |
|-------|---------------------|-------------------|--------|
| auths-core | 433 | **0** | Already has deny lints + #[allow] annotations |
| auths-sdk | — | **0** | Already has deny lints, clean |
| auths-id | 586 | **~6** | Needs deny lint + fixes |
| auths-storage | 289 | **~6** | Needs deny lint + #[allow] annotations |
| auths-verifier | — | **1** (FFI) | Needs deny lint + #[allow] |
| auths-cli | — | **~123** (triage needed) | Git hook entry point — the actual risk surface |
| auths-policy | — | **~129** (triage needed) | Needs audit for production vs test |

**Critical finding**: The signing and verification hot paths are **already panic-free**. The actual risk surface is the CLI layer (auths-sign, auths-verify binaries) which runs in git hooks.

## Scope

### In Scope
- Consolidate conflicting clippy config files (clippy.toml vs .clippy.toml)
- Migrate per-crate lint config to workspace-level `[workspace.lints.clippy]`
- Fix ~13 production expect()/unwrap() calls in auths-id, auths-storage, auths-verifier
- Triage auths-cli and auths-policy unwrap calls
- Add `#![allow]` to integration test entry points and test utility crates
- Escalate workspace lints from warn to deny

### Out of Scope
- anyhow → thiserror migration in auths-core/src/api/runtime.rs (separate epic)
- clippy::indexing_slicing enforcement (related but separate concern)
- auths-radicle (excluded from workspace)

## Approach

### Phase 1: Infrastructure (Tasks 1-2)
Set up workspace-level lint configuration without breaking anything.

### Phase 2: Fix violations (Tasks 3-5)
Crate-by-crate, fix or annotate production unwrap/expect calls.

### Phase 3: CLI/Policy triage (Tasks 6-7)
Audit the presentation and policy layers for production panics.

### Phase 4: Lock it down (Task 8)
Escalate from warn to deny once all crates are clean.

## Key Decisions

1. **`ed25519_to_did_key()` at `auths-id/src/identity/resolve.rs:159`**: Change parameter from `&[u8]` to `[u8; 32]` to make it type-safe without Result (avoids semver-breaking signature change)
2. **Mutex poisoning**: Follow auths-core convention — treat as fatal, `#[allow]` with `// INVARIANT: Mutex poisoning is fatal by design`
3. **FFI expect**: `#[allow]` with justification — tokio runtime creation failure at FFI boundary IS fatal
4. **Clippy config**: Merge both files into single `clippy.toml`, verify `disallowed-methods` bans are actually enforced (`.clippy.toml` may be silently overriding `clippy.toml`)

## Acceptance Criteria

- [ ] `cargo clippy --workspace --all-targets --all-features -- -D warnings` passes with zero unwrap_used/expect_used warnings
- [ ] `deny(clippy::unwrap_used)` and `deny(clippy::expect_used)` enforced at workspace level
- [ ] All production `#[allow]` annotations have INVARIANT comments
- [ ] Single consolidated clippy.toml with all disallowed-methods + allow-unwrap-in-tests
- [ ] `cargo nextest run --workspace` passes
- [ ] Integration tests compile without lint errors
