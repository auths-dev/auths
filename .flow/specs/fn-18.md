# fn-18: Schema Stability & Fuzzing

**Priority:** P2 — Enterprise confidence
**Effort:** 1-2 weeks

## Overview

Make implicit JSON contracts explicit by committing JSON Schema artifacts and extending fuzz coverage. Enterprise consumers need schema stability guarantees; fuzzing catches deserialization panics before they reach production.

## Scope

1. Add schemars as workspace dependency and add `JsonSchema` derives to contract types behind feature flags
2. Create `cargo xtask generate-schemas` command
3. Commit schema artifacts to `schemas/` with CI validation
4. Add CESR import fuzz target to auths-keri (only genuinely new target)
5. Add CI nightly fuzz workflow

## Approach

### Schema generation strategy

**Problem:** The primary contract types (`Attestation`, `IdentityBundle`) live in auths-verifier, which must stay minimal-dep for FFI/WASM. `IdentityBundle` doesn't even derive `Serialize`/`Deserialize`.

**Solution:** Add a `schema` feature flag to **auths-verifier** (not auths-id). Behind this feature, add `schemars::JsonSchema` derives. The xtask enables this feature at generation time — WASM builds never enable it. This keeps the default dependency tree clean while allowing schema generation.

For `IdentityBundle` (no Serialize/Deserialize): add `Serialize + Deserialize + JsonSchema` behind the `schema` feature flag. These are only needed for schema generation, not runtime.

Standardize on **schemars 0.8** as workspace dependency. The xtask/telemetry usage of 1.2.1 is for internal types (pairing API docs) — migrate those to 0.8 workspace dep.

### Schema targets

| Type | Location | Notes |
|------|----------|-------|
| `Attestation` | `auths-verifier/src/core.rs:637` | Owned, Serialize/Deserialize already derived |
| `IdentityBundle` | `auths-verifier/src/core.rs:600` | Needs Serialize/Deserialize added behind `schema` feature |
| KERI ICP event | `auths-keri` | Spec-compliant, canonical field ordering |

**Not `CanonicalAttestationData<'a>`** — it has lifetimes. Use `Attestation` (same JSON shape, owned).

**Note:** auths-id already has a `JsonSchemaValidatingEncoder` that does runtime JSON schema validation of attestations. The committed schema artifacts serve a different purpose: external contract documentation and CI drift detection. They complement, not replace, the runtime validator.

### Fuzz strategy

Existing fuzz targets (do NOT duplicate):
- `auths-verifier/fuzz/`: `attestation_parse`, `did_parse`, `verify_chain`
- `auths-policy/fuzz/`: `policy_eval`

New target: `auths-keri/fuzz/fuzz_cesr_import` — random bytes → `import_cesr_to_events(codec, data)` → should never panic. This is NOT a symmetric roundtrip (raw bytes don't round-trip). The fuzz target tests that malformed CESR input is rejected gracefully.

```rust
#![no_main]
use libfuzzer_sys::fuzz_target;
use auths_keri::{CesrV1Codec, import_cesr_to_events};

fuzz_target!(|data: &[u8]| {
    let codec = CesrV1Codec;
    let _ = import_cesr_to_events(&codec, data);
});
```

CI: nightly-only workflow (separate from main CI), 60s per target.

### schemars version reconciliation

- Add `schemars = "0.8"` to workspace `[workspace.dependencies]`
- Migrate auths-core from direct `schemars = "0.8"` to `schemars.workspace = true`
- Migrate xtask and telemetry from `schemars = "1.2.1"` to `schemars.workspace = true` (0.8). Fix any API differences — schemars 0.8→1.x has minor derive attribute changes.

## Risks / Dependencies

- schemars 0.8 for xtask/telemetry requires checking if 0.8 API covers their usage (pairing types with JsonSchema derives)
- Fuzz CI requires nightly Rust (not currently in CI)
- `IdentityBundle` getting Serialize/Deserialize behind feature flag — must not affect default compilation

