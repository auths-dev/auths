# Push HTTP Orchestration from CLI to SDK

## Overview

The CLI layer (`auths-cli`) currently orchestrates HTTP requests directly via `reqwest` in multiple command handlers. This violates the hexagonal architecture: the CLI (presentation layer) owns networking logic that belongs in the SDK (orchestration layer) behind port traits (defined in `auths-core`, implemented in `auths-infra-http`).

**The problem:** If the CLI owns API integration logic, future consumers (mobile SDKs, embedded agents, CI integrations) must rewrite all HTTP orchestration from scratch.

**The model to follow:** `auths-sdk/src/registration.rs` already demonstrates the correct pattern -- accepts `&impl RegistryClient`, builds typed payloads, dispatches on `RegistryResponse.status`, returns `Result<RegistrationOutcome, RegistrationError>`.

## Scope

### In Scope (registry operations -> SDK)

| CLI File | Operation | HTTP Method | Endpoint |
|----------|-----------|-------------|----------|
| `commands/artifact/publish.rs` | Artifact publish | POST | `/v1/artifacts/publish` |
| `commands/id/claim.rs` | Platform claim submission | POST | `/v1/identities/{did}/claims` |
| `commands/device/pair/online.rs` | Session creation + polling | POST + GET | `/v1/pairing/sessions`, `/v1/pairing/sessions/{id}` |
| `commands/device/pair/join.rs` | Session lookup + response | GET + POST | `/v1/pairing/sessions/by-code/{code}`, `/v1/pairing/sessions/{id}/response` |

### Out of Scope (stay in CLI for now)

| CLI File | Reason |
|----------|--------|
| `services/oauth.rs` | GitHub OAuth Device Flow has presentation side-effects (browser open, user code display, terminal prompts). Third-party API, not registry. |
| `services/gist.rs` | GitHub-specific, requires Bearer auth + custom headers. Third-party API. |
| `commands/device/pair/lan.rs` | LAN server uses axum + mDNS (heavy infra deps). Health check is CLI diagnostics. |
| WebSocket in `online.rs` | tokio-tungstenite is transport-level. The SDK will own session polling via HTTP; WebSocket optimization stays in CLI as an optional fast-path. |

### Architectural Decisions

1. **Extend `RegistryClient` vs new traits:** The existing `RegistryClient.post_json()` method already returns `RegistryResponse { status, body }` which is sufficient for publish and claim. For pairing sessions, we need GET-with-path which `fetch_registry_data` provides. No new port traits needed -- extend existing usage patterns.

2. **`AuthsContext` unchanged:** Follow the `registration.rs` precedent -- SDK functions take `registry_client: &impl RegistryClient` as a parameter. No `AuthsContext` builder change needed.

3. **Error types:** One new `thiserror` enum per workflow domain (`PublishError`, `ClaimError`). Pairing already has `PairingError` -- extend it with network variants.

4. **Clock injection:** Any `Utc::now()` calls that move into SDK must accept `now: DateTime<Utc>` as parameter per CLAUDE.md rules.

5. **Incremental migration:** Artifact publish first (simplest, fewest side-effects), then claim, then pairing (most complex). CLI commands thin out one at a time.

## Approach

Each task follows the same 3-phase pattern:
1. **Port layer** -- ensure `RegistryClient` trait methods cover the needed HTTP operations
2. **SDK workflow** -- new function in `auths-sdk/src/workflows/` accepting `&impl RegistryClient`, returning typed `Result`
3. **CLI thinning** -- replace inline HTTP logic with SDK function call, keep only I/O formatting

## Acceptance Criteria

- [ ] `auths-cli/src/commands/artifact/publish.rs` contains zero `reqwest` imports and zero HTTP status code handling
- [ ] `auths-cli/src/commands/id/claim.rs` contains zero `reqwest` imports and zero HTTP status code handling
- [ ] `auths-cli/src/commands/device/pair/online.rs` session creation and HTTP polling use SDK functions (WebSocket may remain)
- [ ] `auths-cli/src/commands/device/pair/join.rs` session lookup and response submission use SDK functions
- [ ] All new SDK functions follow the `registration.rs` pattern: `&impl RegistryClient` parameter, `thiserror` errors, typed outcome structs
- [ ] No `anyhow` in new SDK code
- [ ] No `Utc::now()` in new SDK code -- all time-dependent logic accepts `now: DateTime<Utc>`
- [ ] `cargo nextest run --workspace` passes
- [ ] `cargo clippy --all-targets --all-features -- -D warnings` passes
- [ ] Existing CLI behavior is unchanged (same output, same exit codes)

## Open Questions

1. **Timeout policy:** `publish.rs` uses 30s connect + 60s total, `join.rs` uses NO timeout, `factories/network.rs` uses 30s. Should the SDK standardize timeouts, or should the `RegistryClient` adapter carry its own timeout config?
2. **WebSocket fast-path:** `online.rs` falls back from WebSocket to HTTP polling. If the SDK only exposes HTTP polling, should there be a `PairingTransport` trait later for WebSocket upgrade?
3. **OAuth/Gist migration:** These are deferred. Should a follow-up epic extract them into `auths-infra-http` behind new port traits (`OAuthClient`, `ProofPublisher`)?
