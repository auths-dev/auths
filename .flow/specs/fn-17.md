# OIDC Trust Registry

## Overview

Enterprise security teams need to define boundaries like "GitHub Actions from repo X can only issue deploy:staging attestations." Without this, the OIDC bridge is open-loop — any valid attestation chain can request any capability. The trust registry is the control plane.

## Scope

- Define trust registry types in `auths-policy` (same Batch 1 layer as `ValidatedGlob`, `CanonicalCapability`)
- Trust registry matching logic (provider lookup, repo glob matching, capability intersection)
- Load trust registry from JSON config files in `auths-oidc-bridge`
- Integrate trust checks as a pre-filter in bridge admission (before policy evaluation)
- CLI commands for trust policy management (writes JSON, optionally backed by Git)

## Approach

### Architecture Decision: Types in `auths-policy`, implementation in `auths-oidc-bridge`

Per dependency ordering (`scripts/releases/2_crates.py`):
- `auths-policy` = Batch 1 (no deps) — types and matching logic live here
- `auths-oidc-bridge` = Batch 3 — loading, config, bridge integration live here
- `auths-id` = Batch 5 — **NOT** used for trust registry (too high in dep tree)

Trust registry types only depend on types already in `auths-policy` (`ValidatedGlob`, `CanonicalCapability`), so they belong there.

### Feature gating: `oidc-trust` (separate from `oidc-policy`)

Trust registry and policy evaluation are independent concerns. Gate behind a new `oidc-trust` feature flag rather than reusing `oidc-policy`. Both depend on `auths-policy` for types, but enabling one doesn't require enabling the other. Cargo deduplicates the shared dependency.

### Strong typing: `ValidatedIssuerUrl`

Provider issuer URLs are not raw `String` — they get a `ValidatedIssuerUrl` newtype in `auths-policy/src/trust.rs` with HTTPS validation, matching the codebase's pattern of `CanonicalDid`, `CanonicalCapability`, and `ValidatedGlob`.

### Trust model

- **Deny-by-default**: If a trust registry is configured and no entry matches, the request is denied
- **Provider isolation**: Entries for different providers are never merged
- **Capability intersection**: composed, not duplicated. Trust narrows requested caps (`requested ∩ registry_allowed`), then existing `scope_capabilities()` intersects with chain-granted (`effective ∩ chain_granted`). Three-way intersection without duplicating logic.
- **Issuer exact-match**: `ValidatedIssuerUrl` — HTTPS only, no wildcards
- **Repo glob matching**: Reuse existing `ValidatedGlob` + `glob_match` from `auths-policy/src/glob.rs`
- **Org-pinned repo patterns**: First segment (org) must be literal; only repo name allows wildcards

### Trust checks vs policy evaluation

Trust registry is a **pre-filter** that runs before policy evaluation in the exchange flow:
1. Trust registry check → `BridgeError::ProviderNotTrusted` / `BridgeError::CapabilityNotAllowed` (403)
2. Policy evaluation → `BridgeError::PolicyDenied` (403)
3. Capability extraction + JWT issuance

### Storage strategy

Phase 1 (this epic): Trust policies stored as JSON config files. Bridge reads on startup + optional reload endpoint.
Phase 2 (future): Git-backed storage under `refs/auths/trust-policy/` with signed entries.
